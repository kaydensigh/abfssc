# DIMENSION 2: Bookmarks-as-command-menu

## Summary
The 127 PDF outline bookmarks ARE the application's primary command UI. Because an AcroForm cannot add custom toolbar buttons, ribbon items, or app menus, the outline (bookmarks) pane is the only built-in, always-visible, off-page clickable chrome that can carry a top-level JavaScript /A action, so the author repurposed it into a full nested command tree: a leading band of everyday commands, then collapsible group headings (Enhancements, HELP, Load data file:, ToolBox, The JavaScript Warning, Miscellaneous) plus Date Format, CheckBox Format and Shortcuts groups. Each bookmark runs an inline JS snippet that dispatches into the shared ~76-function gF* library (mostly a single call like gFScK(); or a gFaA-guarded confirm). The design is clever but suffers severe discoverability pain: the guide repeatedly teaches users to even reveal the pane (View > Show/Hide > Navigation Panes > Bookmarks) and to click the flag icon not the name, and warns that readers which let users edit bookmarks (e.g. Foxit) can corrupt the menu.

## Report
## Why bookmarks at all
The guide states the intent flatly. Page 1 (line 17 of usage_guide_text.txt): "The bookmarks in the form are a menu of commands and helpful hints." Page 21 ("Bookmarks - Usage Tips") is explicit that this is a deliberate repurposing: "Some PDF Readers allow you to change bookmarks - that is unwise with this form as our bookmarks function more like a menu of commands rather than links to other parts of the document." This is a workaround for a hard platform limit: an AcroForm/PDF form cannot add custom toolbar buttons, ribbon items, or application menus to Acrobat/Reader. The only built-in, always-visible, persistently-clickable chrome that can carry a JavaScript action (an outline item's `/A` action) is the bookmark (outline) tree. Form-field buttons could run JS too, but they live ON the printed card page and would consume card real estate / print; bookmarks live in the side navigation pane, off the card, and survive scrolling. So the author built the entire command system as outline entries.

## Menu structure (verified against bookmark_titles.txt + the page-35/36 screenshots)
There are exactly **127** outline entries (bookmark_titles.txt is 127 lines; line 127 is the root "ABF System Card"). Note that bookmark_titles.txt is in PDF outline *storage* order, not display order. The page-35 screenshot gives the true **top-level display order**: Status Check, REFRESH, View editable fields, Popup Messages, Text Field Code Basics, **Richness = Rich, Richness = Basic** (note: "Richness = None" is NOT top-level - it lives under Enhancements), Print Preparation, Show Action Areas, then six collapsible GROUP headings: Enhancements, HELP, Load data file:, ToolBox, **The JavaScript Warning** (with single child "Manage Java Warning"), Miscellaneous. The guide (p.37): "Most bookmark group headings show a brief message about the nested bookmarks." Functional categories:

- **Leading common commands**: Status Check, REFRESH, View editable fields, Popup Messages, Text Field Code Basics, Richness = Rich/Basic, Print Preparation, Show Action Areas.
- **Enhancements / Rich-Mode group**: Richness = None, Manage Styles, Rich Mode Functions > Text Colours (Normal, All Black, Suits Red & Black, Spades Blue, Suits Blue.R.O.G., Suits Black.R.B.G.), Fill Colours (Hide/Show/Normal/"Gray and white" Fill), Font = default.
- **HELP help-tree** (a documentation menu, ships "usually fully expanded in the empty form", p.35): HELP, Recent Help, Quick Help, Important (Blue Fill, Text Value Problems, Classification, Signing Hints, Text Overflow...), General (TAB Sequence, HINTS - CheckBoxes, Field Hints, The Display/Editable Fields, Showing Priorities), Abbreviations (A-F, G-N, O-S, S-z), Printing Tips/Problems, TEXT Field Codes > Rich Codes/More Rich Codes/Suit Symbols/Unicode Text/Text Colour Codes/Fill Colour codes/Shortcut codes, Other Programs (For Foxit/Nitro/Adobe/MAC/MPE Users). The IMPORTANT and GENERAL sub-groups exist purely "so you can hide some of the bookmarks in the HELP group" (p.38).
- **Load data file: group**: heading, Load an FDF file, and a nested **_HANDLE with CARE_** sub-group wrapping the destructive Clear & Load FDF.
- **ToolBox group** (p.39): Set Date, Swap Players, Retired Fields (Check/Append/Make/Remove Old Fields), Big X, Trim Leading Spaces, Font Sizes (**Tweak** Font Sizes, Format Font Size 9pt/10pt, Format - Use default), Formatting Finished, Refresh at Open, Encode Unicode, Print the Card, Print Partner's Card.
- **Date Format group**: NoDates, 2017-Sep, 19/09/2017, 19 Sep 17, 2017-09-19, 2017-09-19 17:23 (each saves a code into My_DateFormat).
- **CheckBox Format group**: Big X, All Black, Cross, Tick, Small Black (save a code into My_CheckBoxValues).
- **Shortcuts & More group**: Save shortcut codes, Cancel shortcut codes, Restore default codes, Show extra notes?.
- **Miscellaneous / diagnostics group** ("most useful for the developer of the form", p.41): Expose Java Console, Show Button Info, Show Settings, Log many values, My Notes (and the storage-order neighbour "Make FDF in Log").

## How a bookmark fires: the dispatch layer (evidence from js_blocks.txt)
The actual per-bookmark `/A` action scripts are the 183 JS blocks. They confirm bookmarks are a **thin dispatch layer** over the gF* library. Most are a single function call: js_blocks.txt line 143 `gFScK();` (Status Check), line 164 `gFScE();` (Expose Java Console), line 188 `gFcRFdF();` (Clear & Load FDF), line 506 `gDlRtdF();` (Remove Retired Fields), line 512 `gMkRtdF();` (Make Retired Fields). Some wrap a confirm: block 2 `if(this.numFields>0){if(gFaA(1,2,'Swap Players?','Click <OK> to swap Player details ')==1)gFsP();}`; block 16 reads My_DateFormat then `if(gFaA(1,2,'Set Dates?',...)==1)gFSD();`. Almost every command bookmark routes through two shared helpers: `gFaA` (alert/confirm, called 49x) for the "click OK / long messages split into chunks" interaction, and `gFsH` (show/hide, 64x).

## Bookmark -> function mapping (verified in extracted JS)
- **Status Check** -> `gFScK()` (clean_0_block36.js:2): prints "@HH:MM:ss --Begin: Status Check--", runs file-size/field/annotation checks, and detects a flattened/signed form: `if(this.numFields==0){ mAXbvi='There are no fields in this file - it looks FLATtened/signed. ...MOST bookmarks will do nothing.'}` (lines 58-59). It also offers to append still-populated retired fields. Matches the guide (p.37): "a short report on the state of the form... possibly some other messages about problems it detects."
- **Swap Players** -> `gFsP()` (clean_0_block36.js:2126). Swaps PlayerName_A/B and PlayerNo_A/B, preserving each field's richText, fillColor, borderStyle, strokeColor and lineWidth; an "Optional swap format if active" path keys off `gOzX.indexOf(' SwpBrdr')` (line 2128). Guide p.39: "Swaps the player names and numbers. Optional swap format if active."
- **Expose Java Console** -> `gFScE()` (clean_1_block37.js:1473): body is literally `console.show();` then, only `if(app.viewerType=='PDF-XChange')`, a deliberate null-deref (`var pFXMakeErra=null; var pFXxd=pFXMakeErra.display;`) to provoke an error - a per-reader workaround to force PDF-XChange to surface its console. Guide p.41: "Make the javascript console and log visible. HANDLE WITH CARE."
- **Clear & Load FDF** (under _HANDLE with CARE_) -> `gFcRFdF()` (clean_0_block36.js:462) opens with `gFaA(1,1,'Important WARNING','...We strongly recommend that you SAVE_AS a new file name BOTH BEFORE and ASAP after CLEARing the form... Click <OK> to proceed with CLEAR (Note: it is NOT recoverable)...')` (line 465).
- **Make / Remove Retired Fields** -> `gMkRtdF()` / `gDlRtdF()` (clean_0_block36.js:994/1075) create/remove old field names for FDF import compatibility ("does not work in all PDF reader programs", p.39).
- **Settings store**: the `gFsV(field,value)` helper (clean_3_block35.js:1) maps shorthand codes to a whole family of hidden settings fields: `*C->My_Colours`, `*D->My_DateFormat`, `*F->My_FillColours`, `*I->I_Format`, `*O->My_Font`, `*X->My_CheckBoxValues`, `*Z->My_Codes` (plus My_Rectangles, My_Format*). So Date Format / CheckBox Format / Shortcuts / Text Colours / Fill Colours bookmarks all just poke one of these fields. **Set Date** then reads My_DateFormat (clean_0_block36.js:2400; clean_1_block37.js:222) and the Date_A tooltip points users to "[ToolBox / Set Dates] bookmark... To disable that, use NoDates bookmark."

## Discoverability pain (the menu is hidden by default)
The command surface is invisible until the user manually reveals it. The guide teaches this at least three times: Getting Started (p.7) "At the left of the window - can you see the bookmarks? If not, via menu, try exposing them by enabling <View> <Show/Hide> <Navigation Panes> <Bookmarks>"; repeated verbatim in Bookmarks-Usage-Tips (p.21). Users must also learn an unintuitive click target: "When using a bookmark, it is best to click on the action box/flag rather than the name" (p.21) - reinforced on p.10 "click on the flag beside the named bookmarks to action the bookmark" - because clicking the label can trigger rename/navigation instead of the `/A` action. The author mitigates legibility with color-coding (p.35 Note: "around version 21E15 the colours of several bookmarks were changed for greater clarity in both light gray and dark gray display themes in Adobe products") - clearly visible in the page-36 screenshot, where group headings and danger items (Clear & Load FDF, "_HANDLE with CARE_", "Cancel shortcut codes", "Append Old Fields", "Load an FDF file") are red. Disclosure triangles keep 127 items manageable; the HELP group ships pre-expanded so first-run users see help immediately. There is no in-menu search and ordering is fixed. The menu also doubles as a JS-liveness probe: if Status Check shows nothing, JavaScript is not active.

## Fragility
Bookmarks are user-editable in some readers, which would corrupt the menu: "Do not delete any of the bookmarks!" (p.35); "A small problem with Foxit Reader may be that it allows changing of bookmarks. Please avoid this" (p.44). After signing/flattening, `numFields==0`, so "most of the bookmarks will do nothing" (p.15) and "you can no longer Swap Partners. You can click on Status Check bookmark to check on this status" - the gFScK numFields==0 guard is the form's own self-diagnosis of that dead-menu state. (Two-character code prefixes that earlier versions put on bookmark names were "removed in 2016 update", p.35.)

## Key Mechanisms

### Bookmark outline tree as command menu (127 entries)
- What: 127 PDF outline entries carry inline JavaScript /A actions; a leading band of common commands plus six collapsible group headings (Enhancements, HELP, Load data file:, ToolBox, The JavaScript Warning, Miscellaneous) plus Date Format / CheckBox Format / Shortcuts & More groups form a full nested command tree.
- Evidence: bookmark_titles.txt = 127 lines (root 'ABF System Card' is line 127). Guide p.21: 'our bookmarks function more like a menu of commands rather than links to other parts of the document'; p.1 (line 17): 'The bookmarks in the form are a menu of commands and helpful hints'. Page-35/36 screenshots show the nested, color-coded tree with flag icons and disclosure triangles.
- Why: An AcroForm cannot add custom toolbar buttons, ribbon items, or app menus; the outline pane is the only built-in, always-visible, off-page clickable chrome that can run a top-level JS /A action without consuming printable card space.

### Inline dispatch snippets -> gF* library
- What: Each bookmark's /A action is a tiny JS snippet that calls into the shared ~76-function library, usually one call, sometimes a gFaA-guarded confirm. So the menu is a thin dispatch layer reusing the same functions as field events.
- Evidence: js_blocks.txt line 143 'gFScK();', line 164 'gFScE();', line 188 'gFcRFdF();', line 506 'gDlRtdF();', line 512 'gMkRtdF();'; block 2 'if(this.numFields>0){if(gFaA(1,2,\'Swap Players?\',\'Click <OK> to swap Player details \')==1)gFsP();}'; block 16 reads My_DateFormat then calls gFSD().
- Why: Keeping per-bookmark scripts trivial and centralizing logic in named functions minimizes duplicated minified JS (file-size pressure) and lets bookmarks and field events share behavior.

### gFScK() - Status Check (also a self-diagnosis probe)
- What: Top bookmark; prints a timestamped status report to the console, runs file-size/field/annotation checks, offers to append still-populated retired fields, and detects a flattened/signed form (numFields==0) warning that most bookmarks will then do nothing.
- Evidence: clean_0_block36.js:2-59, incl. line 59 'There are no fields in this file - it looks FLATtened/signed. ...MOST bookmarks will do nothing.'; guide p.37 'a short report on the state of the form'; p.15 'You can click on Status Check bookmark to check on this status.'
- Why: A signed/flattened PDF strips fields and silently disables the whole bookmark menu; Status Check is the only way to surface that dead state and also acts as a JavaScript-liveness probe.

### gFScE() - Expose Java Console
- What: Miscellaneous-group bookmark whose body is console.show(), plus a PDF-XChange-only deliberate null-deref to force that reader to reveal its console/log.
- Evidence: clean_1_block37.js:1473-1480: console.show(); then if(app.viewerType=='PDF-XChange'){...var pFXMakeErra=null; var pFXxd=pFXMakeErra.display;}; bookmark title 'Expose Java Console'; guide p.41 'Make the javascript console and log visible. HANDLE WITH CARE.'
- Why: No dev tools exist in a PDF reader's chrome; exposing the built-in JS console via a bookmark is the only way to surface developer diagnostics, and PDF-XChange needs an error thrown to pop the console.

### gFcRFdF() - Clear & Load FDF, nested under _HANDLE with CARE_
- What: Destructive command that clears the whole form before loading an FDF; opens with a strong unrecoverable-action confirmation urging a SAVE_AS before and after.
- Evidence: clean_0_block36.js:462-465: gFaA(1,1,'Important WARNING','...SAVE_AS a new file name BOTH BEFORE and ASAP after CLEARing... Click <OK> to proceed with CLEAR (Note: it is NOT recoverable)...'); guide p.38 nests it under the '_HANDLE with CARE_' group 'to warn about the nested bookmark(s)'; red-colored in the p.36 screenshot.
- Why: A flat menu has no danger affordance; the author encodes risk structurally with a dedicated warning sub-group + red color + a JS confirm() (gFaA) to gate an irreversible action.

### gFsP() - Swap Players (ToolBox)
- What: Swaps PlayerName_A/B and PlayerNo_A/B, preserving each field's richText, fillColor, borderStyle, strokeColor and lineWidth; supports an optional border/format swap, behind an OK/Cancel confirm.
- Evidence: clean_0_block36.js:2126-2165 (gFsP captures .richText/.fillColor/.borderStyle/.strokeColor/.lineWidth for all four fields; optional path keyed on gOzX.indexOf(' SwpBrdr')); js_all_clean.js:8 / js_blocks.txt block 2 confirm gate; guide p.39 'Swaps the player names and numbers. Optional swap format if active.'
- Why: Swapping supports printing partner's view; exposing it as a ToolBox bookmark (rather than an on-card button) keeps it off the printable card. Print bookmarks now auto-offer the swap, so Print the Card / Print Partner's Card are flagged 'Probably no longer needed' (p.39).

### gFsV() shorthand -> hidden settings fields (preferences store)
- What: Date Format / CheckBox Format / Shortcuts / Text & Fill Colour bookmarks call gFsV with a shorthand code that maps to a hidden My_* settings field; the value persists when the file is saved, acting as the app's prefs store.
- Evidence: clean_3_block35.js:1-26: gFsV switch maps '*C'->My_Colours, '*D'->My_DateFormat, '*F'->My_FillColours, '*I'->I_Format, '*O'->My_Font, '*X'->My_CheckBoxValues, '*Z'->My_Codes; Set Date reads My_DateFormat (clean_0_block36.js:2400); guide p.35 '3 chunk sizes... You set the chosen value as the new default when you save the form'.
- Why: A PDF has no settings dialog or persistent prefs API; bookmarks that poke hidden form fields are the only way to store and restore user preferences inside the saved file.

### Color-coding + disclosure triangles + pre-expanded HELP
- What: Bookmark labels are colored per theme legibility (danger items red, group headings tinted), groups collapse via triangles, and the HELP group ships expanded so first-run users see help immediately.
- Evidence: Guide p.35 Note: 'around version 21E15 the colours of several bookmarks were changed for greater clarity in both light gray and dark gray display themes in Adobe products'; 'The HELP group is usually fully expanded in the empty form'; page-36 screenshot shows red danger items and tinted headings.
- Why: With 127 fixed-order items and no search, color and collapse are the only affordances available in an outline pane to make the menu scannable, signal danger, and onboard new users.

## Limitations & Workarounds
- PDF/AcroForm cannot add custom toolbar buttons, ribbon items, or application menus, so the author repurposes the outline/bookmark pane as the only always-visible clickable JS command surface (guide p.21: 'a menu of commands rather than links').
- Bookmarks are hidden by default and there is no API to force the pane open, so the guide must teach View > Show/Hide > Navigation Panes > Bookmarks (repeated p.7 and p.21).
- Clicking a bookmark label can trigger rename/navigation in some readers, so the guide tells users to click the flag/action-box icon, not the name (p.21; p.10 'click on the flag beside the named bookmarks').
- Bookmarks are user-editable in some readers (notably Foxit), which would corrupt the menu; the form can only ask users not to edit/delete them ('Do not delete any of the bookmarks!', p.35; Foxit caveat p.44).
- Signing or flattening strips fields (numFields==0), silently disabling almost the whole bookmark menu; gFScK detects this and warns but cannot prevent it (p.15; clean_0_block36.js:58-59).
- No in-menu search or reordering and a fixed 127-item list in storage order; mitigated only by collapsible group headings and per-theme color-coding (p.35), plus IMPORTANT/GENERAL sub-groups that exist solely to hide some HELP items (p.38).
- No native settings/preferences store in a PDF, so bookmarks write codes into hidden fields via gFsV shorthand (*D->My_DateFormat, *X->My_CheckBoxValues, *C/*F/*O/*Z/*I -> My_Colours/My_FillColours/My_Font/My_Codes/I_Format); they only persist if the user then saves (preferably under a new name).
- No dev console UI in the reader chrome; 'Expose Java Console' calls console.show(), and for PDF-XChange specifically throws a deliberate null-deref to force the console open (clean_1_block37.js:1473-1480).
- Long alert text cannot be auto-sized by the platform, so the 'Popup Messages' bookmark lets users pick one of 3 chunk sizes for the many bookmark-driven dialogs; default is middle size for small screens (p.10/p.35).
- FDF import from old form versions needs matching field names that no longer exist, so Make/Remove Retired Fields bookmarks create/delete them at runtime - but this 'does not work in all PDF reader programs' (p.39).
- No danger affordance in a flat outline, so an irreversible Clear & Load FDF is wrapped in a dedicated '_HANDLE with CARE_' sub-group + red color + a gFaA confirm (clean_0_block36.js:465).

## User Impact
- First-time users may never find the command UI: with the Bookmarks pane collapsed the whole app looks like a static form; the guide spends several pages teaching how to reveal it.
- Unintuitive interaction: users must click the small flag icon, not the bookmark text, or the command may not fire (or may start a rename) - taught on p.10 and p.21.
- Convenience once revealed: a single side panel exposes every command (status, refresh, richness, colours, printing, FDF import, settings, diagnostics) without cluttering the printable card.
- Safety friction is built in: destructive commands (Clear & Load FDF, Swap Players, Set Date) pop OK/Cancel confirmations via gFaA, reducing accidental data loss but adding clicks.
- Silent failure modes: after signing/flattening, or with JS disabled, bookmarks do nothing with no obvious feedback except Status Check, which itself only works if JS is live.
- Help is gated behind the menu: the HELP bookmark tree is the built-in help system, so users who never open bookmarks miss most guidance; it ships pre-expanded to soften this.
- Settings persistence is opaque: changing date/checkbox/shortcut/colour format via a bookmark only sticks if the user then saves the file (preferably under a new name), and may need re-confirming when swapping between Adobe and Foxit (p.44).
- Editing bookmarks (possible in Foxit) can break the menu, so users on non-Adobe readers risk corrupting the command surface.
- Print bookmarks (Print the Card / Print Partner's Card) are kept but flagged 'Probably no longer needed' since printing now auto-offers the player swap - mildly confusing legacy clutter.

## Surprises
- The 'menu' is genuinely the whole app shell: 127 outline entries cover diagnostics, printing, FDF import, colour/checkbox/date settings and a full HELP tree - far beyond what bookmarks were designed for, a repurposing the guide openly admits (p.21).
- Bookmarks dispatch via inline /A snippets that mostly contain a single function call (gFScK();, gFScE();, gFcRFdF();, gDlRtdF();, gMkRtdF();) - confirming a thin dispatch layer over the shared gF* library, not bespoke per-bookmark code.
- Status Check (gFScK) is partly a self-diagnosis tool: it explicitly detects numFields==0 and tells the user the rest of the menu is dead because the form was signed/flattened.
- 'Expose Java Console' is literally console.show() plus a deliberate null-deref (pFXMakeErra=null; pFXMakeErra.display) gated on app.viewerType=='PDF-XChange' - a per-reader workaround to force that one reader to open its console.
- Danger is encoded structurally AND visually: a '_HANDLE with CARE_' group heading exists solely to wrap the one unrecoverable Clear & Load FDF command, and the page-36 screenshot shows it (and other risky items) colored red.
- Preferences live as codes inside a whole family of hidden fields (My_DateFormat, My_CheckBoxValues, My_Colours, My_FillColours, My_Font, My_Codes, I_Format) written by gFsV shorthand codes - the bookmark menu doubles as the settings system because PDFs have no prefs store.
- The richness bookmarks visually reflect current state: in the p.35 screenshot 'Richness = Basic' is greyed out while 'Richness = Rich' is highlighted, and 'Richness = None' is demoted into the Enhancements group rather than sitting at top level (a correction to the draft's claim it was top-level).
- Bookmark colours were retuned specifically at v21E15 for legibility across Adobe light/dark gray themes - genuine UI polish on an outline pane; and earlier two-character name prefixes were dropped in the 2016 update.
- The Miscellaneous group is candidly described as 'most useful for the developer of the form' (p.41) - debug tools (Log many values, Show Settings, Make FDF in Log) shipped to end users through the same menu.

## Open Questions
- The bookmark titles in bookmark_titles.txt are in PDF outline STORAGE order, not display order; display order and group nesting were reconstructed from the p.35/p.36 screenshots and the Appendix-1 tables. The precise parent/child wiring of all 127 entries was not extracted from the compressed outline object stream.
- Per-bookmark /A scripts were matched to functions via the js_blocks.txt action blocks (e.g. block 2 = Swap Players, block 16 = Set Date, lines 143/164/188/506/512 = Status Check/Expose Console/Clear&Load FDF/Remove/Make Retired Fields) by behavior and string content; the exact /A->block index mapping for every bookmark was not fully enumerated.
- How group-heading bookmarks implement their informational popup (guide p.37: headings 'show a brief message about the nested bookmarks') versus their children's commands - i.e. whether a heading's /A is a gFaA message or a no-op - was not pinned down for each heading.
- Whether the richness/state visual indication (Richness = Basic greyed vs Rich highlighted in the p.35 screenshot) is achieved by static outline colors or re-skinned at runtime by JS; the v21E15 color note implies static authoring, but the state-dependent appearance suggests some runtime restyling.
