# CRITIQUE

## Gaps
- PDF LAYERS / OPTIONAL CONTENT GROUPS (OCGs) are a whole architectural pillar no dimension covered. The Action Areas are NOT just overlay fields - they are real PDF Optional Content Groups whose names start 'Layer AA'. gFbP (clean_0_block36.js:1193-1222) calls this.getOCGs(), iterates them, and for every layer whose name.substr(0,8)=='Layer AA' sets wCXocgs[j].state=false to hide it before printing; gFaP restores via getOCGs() and the gAAsH hash. The open-time gAAsH={} (OpenAction L193) is the Action-Area-state hash. This is the actual mechanism behind 'Show Action Areas' (bookmark 23) and the print-suppression warnings, and it is the form's ONLY use of a genuine PDF visual-layering feature (vs. the field-display-flag tricks used everywhere else). A web rebuild maps this directly to CSS layers/toggles.
- THE CALCULATE-EVENT REACTIVITY MODEL is undocumented by all 7 dimensions. Beyond OpenAction + per-field Format/Validate events, the form has document-level Calculate scripts on settings fields: block26 is the Calculate script of I_Format (the global text-format broadcaster) and block32 is the Calculate script of My_Rectangles (the border/overlay broadcaster). Writing I_Format (via gFsV('*I',...)) fires block26's Calculate, which loops this.numFields and restyles every required text field's fill/border/font/size in one pass; writing My_Rectangles fires block32 which creates/positions the V_/Box overlay fields. OpenAction sets I_THIS='x' then gFcZ(My_FormatA), and pokes I_Format (L544-545: value=hWXdf then 'x'+hWXdf). So the form has THREE reactive tiers - OpenAction (once), per-field Validate/Format (on edit), and document Calculate on My_*/I_* fields (broadcast a setting to all fields) - and REFRESH is just a brute-force way to re-fire the second tier. This is the closest thing to a reactive data-binding system and is central to a faithful rebuild.
- THE TAB/FOCUS WORKFLOW and 'click the side label' imitation-checkbox interaction is only lightly traced. gFTsB (clean_1_block37.js:892) is the OnFocus/MouseDown/MouseUp/OnBlur handler wired onto V_/Box overlay fields (block32 L128-131 sets all four actions: OnFocus->gFTsB(1), MouseDown->gFTsB(13), MouseUp->gFTsB(2), OnBlur->gFTsB(5)); high opt (>7) pops the 'You have touched a special box field' warning and calls gFHsR() to hide all V_/v_ rectangles, optionally setFocus() to the underlying editable field. gRFF (clean_0_block36.js:1299) is the D_-display-field OnFocus handler that swaps display flags (D_->noView, editable->noPrint) and setFocus()es the editable twin - this is the 'click a display field to edit its twin' mechanism. No dimension explained the full overlay-field click choreography (V_/v_/Box fields are dynamically created clickable hot-zones layered over the card).
- THE CONCRETE COLOUR/FILL PALETTES needed for a rebuild were never extracted. OpenAction L705 defines gClrA - the full ordered RGB array backing the !0-!9 text-colour codes and the extended set (BLACK, RED ['RGB',1,0,0], BLUE, FUCHSIA, GREEN ['RGB',0,0.7,0.3], PURPLE ['RGB',0.63,0.13,0.94], CERULEAN ['RGB',0,0.48,0.65], BROWN, DARKBROWN, ORANGE, then grays/white and a second alternate palette). L710 defines gFilA - the fill-colour palette (very pale tints like ['RGB',0.95,1,1] cyan-tint, ['RGB',1,1,0.92] yellow-tint) backing the T-prefixed fill codes in block26. These exact RGB triples are load-bearing for reproducing the card's appearance and no dimension surfaced them.
- THE OVERFLOW INDICATOR mechanism's reliance on a reader Preference (not form code) deserves its own callout as a hard limitation. The '+' overflow marker that warns text is clipped is Acrobat's built-in 'Show text field overflow indicator' preference - the form cannot draw it itself, can only instruct users to enable it (guide p.28, and the page-1 overlay literally lists 'in <Forms>, enable Show_text_field_overflow_indicator'). A web rebuild trivially solves this (CSS overflow detection) - it is a pure platform-limit artifact worth highlighting.
- THE FULL WBF ABBREVIATION SYSTEM as embedded domain knowledge is uncovered. The guide (p.42-43) ships an expanded World Bridge Federation abbreviation list (BAL, HCP, NAT, NF, TRF, RKCB, T/O, FG 'Forcing to game', etc.) plus ABF extensions marked with '*' (ADV, FAV, UNF, XX). The form's HELP bookmark tree has 'Abbreviations A-F / G-N / O-S / S-z' sub-bookmarks (bookmark_titles.txt 74,75,107,108) exposing this in-app. This controlled vocabulary is part of the regulated content a rebuild should embed (e.g. as a glossary/autocomplete).
- THE SOFTWARE COMPATIBILITY MATRIX (guide p.45) is a concrete per-reader capability table that should anchor the 'platform limits' narrative: Adobe Reader DC/XI = R (rich) but NOT F (no FDF file export); Adobe Acrobat Pro/Standard = R+F; Foxit 10.1.3+ = R+F but SWAP uses Basic text; Preview on Mac = AVOID/M. The R/B/P/F/X/V/M legend quantifies exactly which workarounds each reader forces.
- NO dimension covers the gFTAA/Action-Area menu popups or the 'short menu' that the top-left pink rectangle pops (guide p.25: REFRESH or View Editable Fields), nor how clicking the ABF logo re-exposes the A_JN warning - the on-card click affordances (as opposed to bookmarks) are a parallel command surface only partially mapped.

## Cross-cutting insights
- THE UNIFYING THESIS: every odd design choice is a substitute for a missing platform primitive, and they cluster into a small set of 'because a PDF can't X' families. (1) No code-autorun except OpenAction/field-AA events -> OpenAction does all bootstrapping + field-event probes used as a 'did my library compile?' self-test. (2) No module loader/load order -> 78 typeof-undefined idempotency guards. (3) No reactive recompute API -> writing a field value is the ONLY render trigger, so REFRESH re-writes values, settings broadcast via Calculate scripts on My_*/I_*, and even classification repaint is value=value (gFxC). (4) No custom toolbar/menus -> 127 bookmarks as command menu + on-card Action-Area click zones. (5) No rich-text widget -> ! / grave / digit code mini-languages parsed into richValue span arrays. (6) No prefs store -> settings live as hidden My_* form fields that persist in the saved PDF/FDF. (7) No help framework -> help text stored in fields, shown through app.alert (gFsH). (8) No dev console UI -> 'Expose Java Console' bookmark. (9) Can't detect a dead JS engine -> a self-erasing warning field whose persistence IS the error. (10) Incremental-save bloat -> forced Save-As + terse-code/D_-twin split + size nags. The whole app is a catalogue of these.
- THE TWO-FIELD SPLIT (editable code field + D_ display twin) is the single most consequential decision and it radiates everywhere: it exists for file-size (only terse codes are saved/exported), but it forces (a) the REFRESH friction, (b) the 'View Editable Fields' command, (c) the Adobe 'unchanged-field' exposure bug + warning, (d) tooltips living only on the editable twin (so help needs View Editable Fields), (e) the gRFF click-to-edit choreography, and (f) the destructive None<->Rich re-render. Almost every user complaint traces back to this one storage optimization. A web rebuild collapses it to a single value rendered live - eliminating ~5 of the form's pain points at once.
- CAPABILITY IS PROBED AT RUNTIME, NOT ASSUMED, and the form degrades gracefully along a CONTINUOUS richness scale (gRMx 8/5/2/1) rather than a binary. The open sequence writes a test richValue to T_SPANs and reads it back to set gRMx; it fingerprints 5 viewer properties; it self-disables addField (gAfW=false) the moment it throws. This 'feature-detect then downgrade' discipline is unusually sophisticated for a PDF and is the reason the same file behaves acceptably across Adobe/Foxit/Nitro/old readers. It is the PDF-era equivalent of progressive enhancement.
- THE FORM IS AN OFFLINE-FIRST, SCHEMA-VERSIONED DOCUMENT DATABASE. I_LastSave is a provenance/version stamp (gRevC=21E29.gBldT=2115 + timestamp + viewer + identity + refresh count); gRtFL is a declarative rename-migration DSL run at load; retired fields are a self-modifying import shim; FDF is the compact wire format. This is a complete (if primitive) versioning+migration+sync stack implemented entirely inside an AcroForm because there is no server. A web rebuild replaces it with a JSON doc + schema-version field + a migration table seeded from gRtFL - and notably can DROP the surveillance-grade identity/viewer leakage.
- 'PARTNER SYNC' IS A MISNOMER THROUGHOUT - it is cosmetic field-swapping, not data replication. gFsP swaps only PlayerName_A/B and PlayerNo_A/B (+ optional formatting), and print-partner is swap->print->swap-back. The mental model 'two partners share ONE system file; personalization is a view transform' is the correct abstraction and should be stated plainly: there is no second copy, no merge, no conflict resolution.
- EVERYTHING ROUTES THROUGH TWO CHOKEPOINTS: gFaA (every dialog, with long-message pagination + Foxit code remap) and the gFHash type-dispatch (every field-iterating loop). These two are the 'standard library' of the app. Centralizing all I/O in gFaA is why the form can uniformly paginate, hot-cancel, and normalize across readers; centralizing type classification in gFHash + per-callsite ordering strings is why 601 heterogeneous fields can be swept in tight loops. A rebuild's equivalent is a typed field schema + a single notification/modal service.
- THE COMMAND SURFACE IS TRIPLED because no single PDF affordance suffices: bookmarks (off-page, persistent, but hidden-by-default and editable/corruptible), on-card Action-Area OCB layers (visible but consume print space and print-through in Adobe), and field events (inline but tied to specific fields). The author uses all three and spends pages of guide teaching discoverability (reveal the pane, click the flag not the name). This fragmentation is itself a platform symptom.

## Most interesting points
- The form proves JavaScript is alive by DISAPPEARING: the big yellow A_JN field is a self-erasing canary - working JS deletes it on open, so a lingering overlay IS the error message. It even lists the exact 3 Adobe preferences to enable and rescales its own text by counting trailing '!-' markers. (Verified in page-1 image + BLOCK 0.)
- There is no render API - the ENTIRE visual output is a side effect of writing a field's value. REFRESH 'works' by re-assigning every required field's value to itself (with a '^ ' caret so it counts as changed, defeating Adobe's skip-unchanged optimization). A hack to force a hack. Classification recolour, style themes, everything rides on this.
- The Action Areas are the form's ONLY use of real PDF layers (Optional Content Groups named 'Layer AA'); gFbP hides them via getOCGs() before print - yet candidly warns that in Adobe 'the layers will be hidden but the active print process will still print them', so it pops a reprint warning. A workaround that admits it only fully works in Foxit.
- Adobe Reader - the officially recommended free reader - CANNOT export the officially recommended share format (FDF). The workaround 'Make FDF in Log' hand-assembles a valid FDF file one console.println at a time (header, Catalog object, %%EOF) and dumps it to the JavaScript console for the user to copy out by hand.
- The four classification colours are NOT four booleans - they all write a single 'Classification' STRING field; IsClassGreen/Blue/Red/Yellow are virtual call-target names resolved by gFpB via substr(7). Brown Sticker passes '2' (independent toggle), colours pass '1' (radio branch) - the literal parameter encodes the conceptual independence.
- The form ships a declarative schema-migration mini-language INSIDE a bridge card: gRtFL = 'JumpRaiseMinorOther>JumpRaiseMinor UnusualNTOther>UnusualNoTrump ...' with templated prefix~code rules and '!'-delimited ignore tokens, run at load to recognize pre-2020 field names from imported FDFs - backed by runtime addField'd hidden red placeholder fields (a self-modifying import shim that self-disables in read-only readers).
- Every save embeds surveillance-grade provenance in I_LastSave: revision 21E29.build 2115, exact timestamp, viewer type/version/variation, OS platform, AND the user's identity.name/loginName (with *u* fallback) + refresh count. A shared card leaks who/what/where it was last edited.
- Capability is auto-detected, not configured: at open the form writes a test rich-text span to T_SPANs and reads it back; success sets gRMx=8 (full rich), failure 5 or 2, Nitro 5 - a continuous degradation scale. Combined with a 5-property viewer fingerprint and gAfW addField self-disable, it is genuine PDF-era progressive enhancement.
- 'Partner sync' is cosmetic, not data: gFsP swaps only the two PlayerName_A/B + PlayerNo_A/B slots (richText swap gated on rich mode, border swap gated on a SwpBrdr option, format swap behind a Yes/No confirm); print-partner is swap->print->swap-back. One file, two views - no second copy, no merge.
- The dual editable/D_ field split exists purely to shrink the saved file (only terse codes are stored/exported, D_ is regenerable) - but that one optimization causes the REFRESH friction, the 'View Editable Fields' command, the Adobe unchanged-field exposure bug, tooltips-only-on-the-twin, and the destructive None<->Rich rebuild. The single biggest source of user pain traces to a file-size trick.
- Developer tools are smuggled into the shipped product: gVrbL==666 dumps every Doc member; 'Expose Java Console' bookmark throws a deliberate null-deref to force PDF-XChange's console open; the Miscellaneous bookmark group is candidly 'most useful for the developer of the form'. Plus a production typo (gFpB's cosole.println).
- The help text, the autocomplete data, AND the per-field code list are the SAME field property: typing '?' in a field pops its tooltip (event.target.userName) as an alert, and typing a digit string-searches that same tooltip for '\rN : phrase' to expand it. The 2013 combo boxes were destroyed by the 2015 rich-text rewrite and reimplemented as this clumsy modal '?'+digit loop - a documented UX regression.
