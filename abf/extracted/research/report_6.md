# DIMENSION 6: Data model, FDF, versioning & sync

## Summary
The card's entire "data model" is the AcroForm field set (601 unique /T names) organised as three overlapping families: card-content pairs (an editable terse-code field + a D_ display/print twin), embedded settings fields (the My_* family plus the I_LastSave provenance stamp), and dynamically-created hidden "retired" placeholder fields used only for cross-version FDF migration. Persistence/sharing rides on FDF (Forms Data Format) import/export, with a clever console-printed "Make FDF in Log" (gMFiL) workaround because Adobe Reader cannot write FDF. Versioning is captured by a per-save provenance string built from gRevC='21E29' + gBldT='2115' + timestamp + viewer/platform/identity, while a tiny declarative rename-DSL (gRtFL, e.g. "JumpRaiseMinorOther>JumpRaiseMinor") plus retired-field machinery migrates field names renamed in the 2020 update. Partner "sync" is not data replication at all: gFsP cosmetically swaps the two PlayerName_A/B and PlayerNo_A/B slots (and optionally borders and My_FormatA/B) so each partner prints a personalized card from the one shared system file.

## 1. The data model: three AcroForm field families

`field_names.txt` holds **601 unique field names**. There is no external store; the "model" is the set of PDF field values, which is exactly what a saved PDF and an exported FDF persist.

**(a) Card-content pairs (rich fields).** Each rich field is split into an *editable* field holding terse codes (`Open1C`, `BasicSystem`) and a `D_`-prefixed *display* twin (`D_UnusualNT`, `D_UnusualNoTrump`, field_names.txt:283-286). Guide p.16 is precise: "rich text fields have twin PDF fields sitting in the same position... One is used for data entry/editing **and exporting to FDF file. Only this one has a hint** with relevant bridge details. The other is used for showing the encoded value on the screen and for printing... The REFRESHed display shows the D_* fields." Critically for a web rebuild: **only the editable field is exported**, and the D_ twin is regenerable from the editable value via REFRESH, so a clean model only needs the editable side.

**(b) Embedded settings/metadata (My_* family).** A dedicated `My_*` family stores user configuration inside the same field set as the card data, so it travels in both the saved PDF and the FDF for free (field_names.txt:340-352): `My_CheckBoxValues, My_Codes, My_Colours, My_DateFormat, My_FillColours, My_Font, My_FormatA, My_FormatB, My_Options, My_Rectangles, My_Richness, My_Styles, My_Symbols`. `gFsV` (clean_3_block35.js:1-26) gives terse aliases that confirm these are first-class slots: `*C`âMy_Colours, `*D`âMy_DateFormat, `*F`âMy_FillColours, `*I`âI_Format, `*O`âMy_Font, `*X`âMy_CheckBoxValues, `*Z`âMy_Codes.

**(c) Provenance/version metadata.** `I_LastSave` is stamped on every WillSave (clean_1_block37.js:1603-1634):
`kMXs = gRevC+'.'+gBldT+' ' + util.printd('yyyymmddHHMM') + ' f:'+app.formsVersion + ' v:'+app.viewerVersion + ' T:'+app.viewerType + ' V:'+app.viewerVariation + ' o:'+app.platform + ' n:'+identity.name + ' R:'+gNumR`.
A detail the draft missed: `identity.name` has a **fallback chain** â if empty it tries `identity.loginName`, then `?`; if undefined it writes `*u*` (lines 1612-1631). I confirmed the embedded constants directly (`_openaction_clean.js`): **`gRevC='21E29'`, `gBldT='2115'`, `gNumR=0`** â so the stamp begins `21E29.2115 yyyymmddHHMM ...`. (The *guide* PDF itself is older, "Rev.21D24", so the guide trails the form.) `A_JN` carries the JavaScript-disabled warning; on save, if its length >600 it is set `display.noPrint` (clean_1_block37.js:1636-1647), gated by `gOzX.indexOf(' HideJ')`. `Date_A` is the user-facing revision field; its tooltip (field_tooltips.txt:285) reads: "[Optional] Your revision code... Choose between date or version number/code. If date - may be set using [ToolBox / Set Dates] bookmark or clicking on the MyRev. side label." `Classification`/`IsBrownSticker` (field_names.txt:32 region, :321) are regulatory metadata.

## 2. Persistence & sharing via FDF

Guide p.31 states the intent: "FDF is short for 'Forms Data Format'... hold the contents of user entered data fields... a compact way of storing values... also allow moving your system details to a future version of the PDF form. Our FDF files are much smaller (at about 30kB) than the filled PDF form." This is *why* the rich/D_ split exists (p.6: "Rich fields now use one field for editing & export, & printing. This is to help minimize the size of the filled form").

**Import â `gFgFdF`** (clean_0_block36.js:779-970), the "Load an FDF file"/"Clear & Load FDF" engine, calls native `this.importAnFDF()` (no-arg â OS file picker at line 889; path-arg â direct at line 874). It logs `I_LastSave` before/after, supports a queued `gNeXtFdF` list (`'; '`-separated names; trailing `*` = "more to load"; `_*F=` prefix sets an override folder `sBXzF`, lines 812-815), prompts Yes/No/Cancel per file (lines 821-822), and wraps `this.documentFileName` in try/catch (lines 862-868), falling back to deriving the folder from `this.path` minus filename length. After import it scans the retired-field hash `gOfPH` and offers to **append old values to the adjacent new field** (lines 928-961), or to merely log them.

**Export â the Adobe Reader workaround.** Guide p.31: exporting FDF "is **not available in any version of Adobe Reader**" (only Foxit Reader, Acrobat Pro/Standard, and some others). The bookmark "Make FDF in Log" â **`gMFiL`** (clean_3_block35.js:60-283) prints FDF text to the **JavaScript console** instead of writing a file. Two modes selected by the option string:
- `*_XFS_`: tries native `this.exportAsFDFStr({bAllFields:1,bFlags:1})` if it exists (lines 95-98), but first scans for any text field whose value >2000 chars and, if found, falls back to exporting only a **hard-coded sample set** (line 112): `A_HELP,A_JN,BasicSystem,Classification,Compete1NT_1,DefenceStrongC_2,I_LastSave,IsBlackwood,My_CheckBoxValues,My_Rectangles,Open1C,Resp1C_1H,...`. It then re-inserts line breaks for legibility (`'Fields[<<'`â`'Fields[\r<<'`, `/>><</g`â`'>>\r<<'`, `Kids[<<`â`Kids[\r<<`).
- `*_FiL`: hand-builds a valid FDF by iterating `this.numFields`, escaping `\ ( )`, and **token-encoding whitespace**: `\r\n`â`<<R>>`, `\r`â`<R>`, `\n`â`<N>`, `\t`â`<T>` (lines 232-238; note the `\r\n` pair is its own token, a nuance the draft flattened). It `console.println`s `%FDF-1.2 / 1 0 obj / <</FDF<</Fields[ ... ]>>/Type/Catalog>> / endobj / trailer / <</Root 1 0 R>> / %%EOF` (lines 247-279), prefixing each field with a category tag (`_z_`,`_o_`,`_w_`,`_t_`,`_h_`,`_a_`,`_i_`,`_S_`) derived from `gFHash[fn]` and name prefix. It even emits synthetic `aaNote_*` pseudo-fields documenting the parameters and how many fields were skipped.

## 3. Cross-version migration (the 2020 rename)

Guide p.6: "many fields have new names. **The old field names are also included to help with importing FDF files from earlier versions.**" Mechanism:
- **`gRtFL`** is a declarative rename-DSL string. I located its actual value (`_openaction_clean.js`) â an open question in the draft: `'Player~.>_ Resp~.O>_O Resp!ons!Oth!._~._>_ Resp1NT2COther>Resp1NT2CStyle JumpRaiseMinorOther>JumpRaiseMinor JumpRaiseMajorOther>JumpRaiseMajor UnusualNTOther>UnusualNoTrump Over1NTInterfMore>Over1NTInterf'`. So rules are either a simple `OLD>NEW` pair, or a templated `prefix~oldcode>newcode` (with `!`-separated ignore-tokens that suppress matching field names containing them).
- **`gMRFL`** (clean_1_block37.js:27-102) parses each token: for `~` rules it scans every field name whose prefix matches `jAXrpx`, skips names containing the up-to-three ignore tokens, then substitutes `newcode`â`oldcode` within the name and registers via `gARFE(oldName,newName)`.
- **`gARFE`** (clean_0_block36.js:972-991) populates parallel arrays `gOfNA`/`gOmNA` (old/new names) and hashes `gOfPH` (oldâold-field-object), `gOmPH` (oldânew-field-object), tagging `gFHash[old]='o'`.
- **"Make Retired Fields"/`gMkRtdF`** (clean_0_block36.js:993-1072) dynamically `this.addField`s the old names as **hidden, readonly, red-bordered** placeholders parked in a column near rect xâ2 (`borderStyle=border.s; strokeColor=color.red; lineWidth=1; textFont=font.Helv; textSize=8; readonly=true; display=display.hidden`), so importing an old FDF lands values somewhere recognized. It is gated by `gAfW` ("add-field works"), set false the moment `addField` throws or returns the wrong name â i.e. it self-disables in readers without editing rights.
- **"Remove Retired Fields"/`gDlRtdF`** (clean_0_block36.js:1074-1118) deletes them, but only when empty (`value.length<=1`) and still at the parked rect (`rect[0]<=2`), after a confirm.
- **"Append Old Fields"/`gFaRFs`** (clean_1_block37.js:104-147) concatenates each retired value onto its new partner using the separator **`' !t!1 '`** (note the surrounding spaces â the draft wrote `!t!1`), after stripping placeholder sentinels (`' '`, `'^ '`, `'!n '`, `'!= '`, `'!+'`, `'!-'`). Guide p.39 warns both addField/removeField "do not work in all PDF reader programs."

## 4. Partner sync â one system, two personalized cards

`gFsP` (clean_0_block36.js:2125-2214) swaps `PlayerName_A`â`PlayerName_B` and `PlayerNo_A`â`PlayerNo_B`, copying `value`, `fillColor`, `richText` (**only if `gRMo==8`**, i.e. rich mode), and â only if `gOzX` contains `' SwpBrdr'` â `borderStyle`/`strokeColor`/`lineWidth`. A nuance the draft glossed: the `My_FormatA`â`My_FormatB`/`I_Format` swap is **not unconditional** â it fires only in rich mode and after an `app.alert` "Swap format?" ## 6. What a clean web data model needs

(1) a flat keyâvalue map of **only the editable** card fields (drop D_ twins; render display from the code); (2) a typed **settings object** (colours, fill colours, font, checkbox style, date format, richness, rectangles, format A/B, custom shortcut codes); (3) **player identity** as a 2-element array with a "primary player" pointer instead of physically swapping values; (4) **classification/brown-sticker** as enumerated regulatory metadata; (5) a **provenance block** (schema version, build, timestamp, author) replacing `I_LastSave`/`A_JN` â and note it currently leaks `identity.name`/viewer/OS; (6) a JSON document format with an explicit **schema-version field** and a server-side **rename migration table** (seeded directly from `gRtFL`'s rules) replacing the retired-field placeholder shim â done once on load, not via mutating hidden fields.

## Key Mechanisms

### gMFiL ("Make FDF in Log")
- What: Generates an FDF document and writes it to the JavaScript console (console.println). Mode *_XFS_ uses native exportAsFDFStr (with a 2000-char-value guard that falls back to a hard-coded sample field set); mode *_FiL hand-builds %FDF-1.2...Type/Catalog...%%EOF by iterating this.numFields, escaping backslash/parens and token-encoding whitespace (\r\n->'<<R>>', \r->'<R>', \n->'<N>', \t->'<T>'). Emits synthetic aaNote_* fields documenting the parameters/skip counts.
- Evidence: clean_3_block35.js:60-283; lines 96-98 exportAsFDFStr({bAllFields:1,bFlags:1}); line 112 sample-field fallback list; lines 232-238 token encoding; lines 247-279 console.println('%FDF-1.2')...'%%EOF'; guide p.31 'not available in any version of Adobe Reader'
- Why: Adobe Reader (the recommended free viewer) cannot WRITE FDF, but JS can always println to the console; dumping copy-out-able FDF text is the only export path without Acrobat Pro/Foxit. A pure PDF-platform-limitation workaround, added ~rev 21E09/21E15 (guide p.31).

### gFgFdF (Load / Clear & Load FDF)
- What: Imports FDF via native this.importAnFDF() (no-arg = OS picker, path-arg = direct load). Supports a queued '; '-separated filename list gNeXtFdF (trailing '*' = more-to-load; '_*F=' token = override folder), per-file Yes/No/Cancel prompts, logs I_LastSave before/after, wraps documentFileName in try/catch, then offers to append retired-field values to their new partners.
- Evidence: clean_0_block36.js:779-970; this.importAnFDF(sBXfldr+sBXnf) line 874; this.importAnFDF() line 889; 'Use this FDF file?' gFaA prompt lines 821-822; _*F= folder token lines 812-815; documentFileName try/catch lines 862-868; append loop 928-961
- Why: importAnFDF is the only built-in data-load primitive a PDF form has; the queue + prompts enable the author's preferred 'layer FDF over the current form' workflow (guide p.31) and let multiple cards/overlays be applied in order.

### gRtFL rename-DSL + gMRFL/gARFE + gOfNA/gOmNA/gOfPH/gOmPH
- What: gRtFL is a space-separated rename-map ('OLD>NEW' or 'prefix~oldcode>newcode' with '!'-delimited ignore-tokens). gMRFL parses it, scanning every field name and registering old->new matches via gARFE, which fills parallel name arrays and old->field-object hashes so the form recognizes pre-2020 names from imported FDFs.
- Evidence: gRtFL='Player~.>_ Resp~.O>_O Resp!ons!Oth!._~._>_ Resp1NT2COther>Resp1NT2CStyle JumpRaiseMinorOther>JumpRaiseMinor JumpRaiseMajorOther>JumpRaiseMajor UnusualNTOther>UnusualNoTrump Over1NTInterfMore>Over1NTInterf' (_openaction_clean.js); gMRFL parser clean_1_block37.js:27-102; gARFE clean_0_block36.js:972-991 (gOfPH/gOmPH/gFHash[old]='o'); guide p.6
- Why: The 2020 rename would have broken FDF interop; a compact declarative map + dynamically materialized placeholder fields lets one form ingest data exported by any prior version without re-keying. Essentially an embedded schema-migration DSL inside a bridge card.

### Make/Remove Retired Fields (gMkRtdF / gDlRtdF)
- What: gMkRtdF dynamically this.addField()s hidden, readonly, red-bordered text fields named with OLD field names (parked near rect x=2) so imported old-version FDF values have a landing slot; self-disables via gAfW if addField throws or returns the wrong name. gDlRtdF removes them, but only when empty and still parked.
- Evidence: clean_0_block36.js:993-1072 (this.addField(wBXon,'text',0,wBXrect); strokeColor=color.red; readonly=true; display=display.hidden; gAfW guard lines 1038-1043); gDlRtdF 1074-1118 (rect[0]>2 and value.length>1 skip); guide p.39 'do not work in all PDF reader programs'
- Why: importAnFDF only fills fields that ALREADY exist; to receive a renamed field's value you must first materialize a field with the old name. That needs document-editing rights, hence the 'not in all readers' caveat and the gAfW self-disable.

### I_LastSave provenance stamp
- What: On WillSave, writes one string: gRevC.gBldT + yyyymmddHHMM timestamp + Adobe formsVersion/viewerVersion/viewerType/viewerVariation + OS platform + saving user's identity.name (with loginName/'?'/'*u*' fallbacks) + refresh counter gNumR.
- Evidence: clean_1_block37.js:1603-1634: kMXs=gRevC+'.'+gBldT+' '+util.printd('yyyymmddHHMM',new Date())+' f:'+app.formsVersion+...+' n:'+kMXnn+' R:'+gNumR; fallback chain lines 1612-1631; constants gRevC='21E29', gBldT='2115', gNumR=0 in _openaction_clean.js
- Why: No app database exists to log to; embedding a provenance/version line in a field is the only way to record which form revision + reader + user last touched the card, for support and migration decisions. Stamp begins '21E29.2115 ...'.

### gFsP (Swap Players) + gFxPx (Print Partner's Card)
- What: gFsP swaps PlayerName_A/B and PlayerNo_A/B (value + fillColor always; richText only if gRMo==8; border style/color/width only if gOzX has ' SwpBrdr'). In rich mode it also offers, via an app.alert confirm, to swap My_FormatA/B and I_Format. gFxPx wraps swap->print->swap-back.
- Evidence: gFsP clean_0_block36.js:2125-2214 (this.delay=true; richText gated by gRMo==8 line 2158; SwpBrdr conditional 2128; My_FormatA/B swap behind app.alert 'Swap format?' 2199-2210); gFxPx 2216-2221 'gFsP();gFPTC();gFsP();'; guide p.20/p.32
- Why: A partnership maintains ONE system file; swapping just the two player slots (and optional per-player formatting) yields personalized printouts without duplicating card data. Partner 'sync' is cosmetic, not data replication.

### My_* settings fields + My_Codes shortcut engine + REFRESH nudge
- What: User config (colours, fill colours, font, checkbox values, date format, richness, rectangles, format A/B, custom shortcut codes) lives as ordinary AcroForm fields. The grave-accent ('`'=gCdS) substitution reads runtime hash gCdH seeded by gFrC defaults / My_Codes. REFRESH re-applies every setting by nudging each My_* field's value (append a char, then restore) to re-fire its format script.
- Evidence: field_names.txt:340-352 (My_* family); gFsV aliases clean_3_block35.js:1-26 (*Z->My_Codes); gCdH lookup clean_2_block20.js:52-82; gCdS='`' and gFrC defaults clean_3_block35.js:37-58; REFRESH nudge clean_0_block36.js:2386-2403; bookmarks 'Save/Cancel/Restore default codes' bookmark_titles.txt:32-34
- Why: Storing settings in fields makes them persist in both saved PDF and exported FDF with zero extra plumbing, and lets users pre-seed them via FDF. The value-vs-defaultValue pair implements Save/Restore of custom codes; the nudge broadcasts a single setting change across a passive field set.

## Limitations & Workarounds
- Adobe Reader cannot WRITE FDF (only Foxit Reader, Acrobat Pro/Standard, some others can). Workaround: 'Make FDF in Log' (gMFiL) PRINTS a valid FDF document to the JS console for the user to copy out. Guide p.31 confirms export 'is not available in any version of Adobe Reader' and that this method was added around rev 21E09/21E15.
- importAnFDF only fills fields that already exist, so a renamed field can't receive an old-version value. Workaround: dynamically addField hidden/readonly/red placeholder fields with the OLD names ('Make Retired Fields'), import, then 'Append Old Fields' (gFaRFs) to merge them onto the new field with a ' !t!1 ' separator. addField/removeField need editing rights and 'do not work in all PDF reader programs' (guide p.39); the code self-disables this path via the gAfW flag the moment addField fails.
- The rich/D_ twin split exists largely to minimize saved-file size: only the editable field is exported to FDF, and the D_ display twin is regenerated by REFRESH (guide p.6 'one field for editing & export, & printing... to help minimize the size of the filled form'). A workaround for PDF bloat when rich text is stored.
- PDF forms cannot add custom toolbar buttons, so EVERY data operation is exposed as a bookmark acting as a command menu: 'Make FDF in Log', 'Load an FDF file', 'Clear & Load FDF', 'Swap Players', 'Print Partner's Card', 'Save/Cancel/Restore default codes', 'Make/Remove/Check Retired Fields', 'Append Old Fields', 'Status Check', 'Log many values' (bookmark_titles.txt).
- FDF readability is poor: 'Some PDF Readers export hexadecimal encoded field names and/or values and may omit whitespace between the fields' (guide p.31). The hand-built gMFiL FDF deliberately re-inserts line breaks ('Fields[<<'->'Fields[\r<<', '>><<'->'>>\r<<', 'Kids[<<'->'Kids[\r<<') and token-encodes \r\n/\r/\n/\t as <<R>>/<R>/<N>/<T> so the dumped text stays legible AND re-importable.
- gMFiL's native exportAsFDFStr path guards against fields whose value exceeds 2000 chars (it would bloat/break the export); if any is found it abandons a full export and dumps only a hard-coded ~15-field sample set (A_HELP,A_JN,BasicSystem,...). So even the 'native' path is not guaranteed complete.
- Digital signing / flattening / saving to signing internet sites can break the form's JavaScript and bookmarks. The recommended read-only-share path is save-as -> Swap Players -> sign -> send; the guide repeatedly discourages signing/flattening (p.32).
- this.documentFileName is unavailable in some readers; gFgFdF wraps it in try/catch and falls back to deriving the folder from this.path minus filename length (clean_0_block36.js:862-871).
- The usage guide itself (PDF text says 'Rev.21D24') lags the form (gRevC='21E29'), so guide page references to features like the FDF-in-Log method point forward to an 'Advanced Guide' not yet written ('please contact the author').

## User Impact
- Two partners maintain one system file and each prints a personalized card (own name/number first) via 'Swap Players'/'Print Partner's Card'. Convenient, but a permanent swap mutates the file, so they must Save As to preserve the canonical ordering; the guide's share flow explicitly says 'save with a new name' first.
- Upgrading to a new form version is meant to be 'export FDF -> open new form -> Load FDF', but Adobe Reader users can't export FDF normally and must use the awkward 'Make FDF in Log' -> copy-from-console -> paste-to-file procedure, or obtain Foxit/Acrobat Pro. The guide concedes this is 'not very user friendly.'
- Importing a pre-2020 card triggers a multi-step migration: Load FDF, then prompts to 'Append Old Fields', plus Trim Leading Spaces, set 'Big X', re-pick fonts/colours, and REFRESH (guide Appendix). Powerful but error-prone; some old values are cloned into multiple new fields the user must hand-tidy.
- Settings (colours, fonts, checkbox style, date format, custom shortcut codes) ride inside the same field set, so they survive save/FDF round-trips and can even be pre-seeded by sending an FDF - handy for a teacher distributing a base config, but it means a shared FDF silently carries the sender's display preferences.
- FDF files are ~30kB vs a much larger filled PDF, so they're far easier to email - but the recipient must already have the matching blank form (from their national web site) and a compatible reader, adding coordination.
- The I_LastSave stamp records the user's identity name (or login name), Adobe viewer type/variation, forms version, and OS platform on every save; useful for support diagnosis but it means a saved/shared card leaks who/what/where it was last edited.
- Because everything is bookmark-driven and JavaScript-dependent, opening in a browser or a non-Adobe/Foxit reader degrades or disables import/export/swap and the retired-field shim entirely; the form warns not to open in a browser.

## Surprises
- The actual gRtFL rename rules (an open question in the draft) are short and recoverable from _openaction_clean.js, e.g. 'JumpRaiseMinorOther>JumpRaiseMinor', 'UnusualNTOther>UnusualNoTrump', and templated forms like 'Player~.>_' and 'Resp!ons!Oth!._~._>_' - a genuine embedded schema-migration mini-language with '!'-delimited ignore-tokens.
- The version constants are literal in the open-action init: gRevC='21E29', gBldT='2115', gNumR=0, so I_LastSave begins '21E29.2115 ...'. The usage-guide PDF, by contrast, still says 'Rev.21D24' - the docs trail the code.
- 'Make FDF in Log' literally hand-assembles a syntactically valid FDF file (header, Catalog object, trailer, %%EOF) one console.println at a time, even emitting synthetic 'aaNote_*' pseudo-fields that narrate the export parameters and how many default-valued/empty fields were skipped.
- The whitespace token encoding treats \r\n as its own token ('<<R>>'), distinct from lone \r ('<R>') and \n ('<N>') - a subtlety needed so Windows CRLFs round-trip exactly through the console-dump-and-reimport path.
- Even the 'native' exportAsFDFStr path is not trusted for big fields: a single value >2000 chars makes gMFiL abandon the full export and dump only a fixed ~15-field sample, so users with long notes silently get a partial FDF.
- To receive renamed fields from old FDFs, the form mutates its OWN AcroForm at runtime - addField()ing hidden RED readonly placeholder fields named with the OLD names - then merges them; an import shim built by self-modification, which self-disables (gAfW=false) in readers without editing rights.
- Partner 'sync' isn't replication; gFsP cosmetically swaps two player slots, and the My_FormatA/B swap is gated behind an interactive 'Swap format?' app.alert and only in rich mode (gRMo==8) - more conditional than the draft implied.
- REFRESH re-applies settings by 'nudging' each My_* field (append a char, then restore the value) to re-fire its format script - exploiting AcroForm's per-field recalculation as a poor-man's reactive update across an otherwise passive field set.
