System Card · Reverse-Engineering Notes

A PDF that thinks it’s an application

How the ABF Standard System Card—a two-page bridge convention card—became a 148 KB JavaScript program living inside an Acrobat form, and why almost every strange decision in it is a workaround for something a PDF simply cannot do.

♠ ♥ ♦ ♣

01What this artifact actually is

An entire application, smuggled into a two-page Acrobat form

On paper, the ABF Standard System Card is one folded A4 sheet on which a bridge partnership writes down its bidding and play agreements. The file in abf/ABF_Card_FORM.pdf reproduces that sheet—but behind the two visible pages sits a complete software product: roughly 148 KB of minified JavaScript spread across 183 separate /JS action blocks, defining a library of about 76 functions that drive 601 form fields, a 127-entry command menu, a rich-text rendering engine, a settings system, a schema-migration layer, and an FDF data-exchange format. It even ships with its own 53-page Usage Guide because, as the guide cheerfully admits on page 1, “The PDF form is quite different from other PDF forms you may have used.”

The single most useful lens for understanding the whole thing is this:

The thesis

A PDF reader gives JavaScript almost none of the primitives an app developer expects—no toolbar, no custom dialogs, no module system, no “re-render” call, no preferences store, no reliable rich-text widget, no guarantee the code will even run. Nearly every peculiar feature of this form is a substitute for one of those missing primitives. Read it that way and the oddities stop being quirks and become a catalogue of clever—and sometimes desperate—workarounds.

The form is the work of one author—the footer reads “PDF Form Rev. 21E29 by RoL”, and the guide names him as Rex Livingston. His initials are fossilised throughout the code: virtually every function parameter is named with a trailing RoL (e.g. gFsH(qDXRoL)). The naming scheme is terse and consistent—helper functions are prefixed gF, with gR/gM/gS/gT families for rich/make/set/toggle operations, and locals carry three-character scramble prefixes (hSX, qDX, yEX…) to stay collision-free after minification.

02The domain it must reproduce

A regulated full-disclosure document for a bridge partnership

In duplicate bridge, opponents are entitled to know how you bid and play. A convention card (or “system card”) is the standardised sheet a pair fills in so the opposition and the director can read its agreements at the table. The ABF’s card layout—the numbered sections, the order, the classification scheme—is itself the regulated artifact. A faithful rebuild therefore has to reproduce the same sections and the same classification model, not merely “a bridge form.”

Two pages, folded into a booklet

The PDF has exactly two pages, mirroring the two sides of the folded paper card. PDF page 1 is the Outside (the masthead plus sections 1–7), and PDF page 2 is the Inside (sections 8–10)—the same numbering the guide uses. A subtle but important layout fact: on the outside page the printed front cover (masthead + section 1) sits in the right-hand column, not the left. The guide explains why (p.2): “It simplifies printing. When printed, we fold the card in the middle and then the booklet opens at the right.” On screen the reading order is therefore non-linear—a print-layout constraint leaking into the editing experience.

Classification: one enum and one independent flag

The defining piece of metadata is the system Classification—a regulatory colour bucket (Green / Blue / Red / Yellow) shown as a large coloured circle in the masthead—plus an independent Brown Sticker flag. A revealing implementation detail: these are not four booleans. There is a single string field named Classification; the names IsClassGreen/Blue/Red/Yellow are virtual call targets that don’t exist as fields at all. Clicking a colour runs a one-liner like gFpB('1','IsClassGreen'), and gFpB reads the suffix after IsClass, then sets (or toggles off) the one Classification string—genuinely radio-like behaviour from a single value. The Brown Sticker is passed as '2' instead of '1'; that literal parameter encodes its conceptual independence from the colour.

Deliberately out of scope

The form stores and renders the classification colour but does not embed what each colour means. The guide (p.24) defers to “the latest national Regulations when classifying your system.” A web rebuild must source the Green/Blue/Red/Yellow semantics, and the WBF/ABF abbreviation vocabulary (FG = Forcing to game, TRF = transfer, RKCB, T/O… shown in the guide’s Abbreviations appendix), from current ABF regulations—they are not in the file.

03The runtime architecture

Three reactive tiers, two chokepoints, and 78 idempotency guards

A PDF can run code in only a few situations: once via the document’s OpenAction, on field events (validate / format / focus / mouse / calculate), and on document lifecycle hooks (WillPrint, WillSave, WillClose). The form’s whole control flow is built from exactly those hooks, arranged into three tiers.

Tier 1OpenAction bootstrapruns once, at open

A single, uncompressed document-level script (~26 KB) declares every global, defines the lowest-level primitives the rest of the code only ever calls (gFaA, gFpB, gMDH, gFXa…), builds the gFHash field-type map for all 601 fields, probes the reader’s capabilities, and kicks off initialisation via gFiF('AtOpenUp').

▼
Tier 2Per-field Validate / Format eventson every edit

Each rich field’s own actions expand shortcut codes (Validate) and build the rendered rich-text spans (Format). This is the render engine. Writing a field value is the only way to trigger it.

▼
Tier 3Document Calculate scripts on settings fieldswhen a setting changes

Hidden My_* / I_Format fields carry Calculate scripts that, when written, loop over all fields and broadcast a style/colour/border change across the whole card—the closest thing here to reactive data-binding.

Because the 76-function library is duplicated across many of the 183 action strings (any one of them might be the “first” to execute), every function is wrapped in an idempotency guard—there are exactly 78 of them:

if(typeof gFsH==='undefined'){
  function gFsH(qDXRoL){ /* … installs itself on first use … */ }
}

There is no module loader and no guaranteed load order, so this lazy self-installation is how the library reliably exists by the time any field, bookmark, or hook needs it. (The cost: the library is repeated many times over, which is part of why the file is large.)

Two chokepoints do most of the work

  • gFaA(type, icon, title, message) — the universal alert/confirm wrapper, called 49×. Every dialog in the app flows through it, which is how the form can uniformly paginate over-long messages into sequential “click OK to proceed” alerts (Acrobat dialogs grow downward and push the OK button off-screen), offer an auto-OK “hot cancel”, and normalise return codes across readers—Foxit returns 6/7 for Yes/No where Acrobat returns 4/3, so gFXa remaps them.
  • gFHash + per-call-site ordering strings — every field is mapped to a one-character type code (T text, R display-twin, B checkbox-display, V overlay, S setting…). Hot loops over all 601 fields branch by indexOf into a position string such as 'T|>RLyzBQxKvwoiSjV'. Intriguingly there isn’t one canonical ordering—different functions re-tune the string (gFCX uses 'T|>RoOSijVxBKQvwyz') so their own index-range tests line up.

The code is packed as PDF string literals

One detail only visible when you decompress the file: the minified JavaScript is stored hard-wrapped, with each wrap point escaped using a PDF line-continuation sequence (a backslash immediately followed by an end-of-line). So var x=0; is physically stored as va\ + CR + r x=0;. The reader strips the \+EOL to rebuild the source. It’s a reminder that this code wasn’t written in a PDF—it was compiled into one.

04Rich fields & the code languages

The twin-field trick, three stacked shortcut systems, and a render engine made of strings

The card’s cells are tiny and its content is dense, so the form needs two things a plain PDF field can’t offer: a way to type very little and display a lot, and a way to show coloured suit symbols and formatting. Both are solved by the same architecture.

The editable / D_ twin split

Every rich entry is actually two overlapping fields in the same position: an editable field holding terse codes (e.g. BasicSystem = “1”), and a D_-prefixed display twin (D_BasicSystem) holding the pre-rendered rich text that is shown and printed. There are 245 such D_ twins. The guide (p.6) states the reason plainly: “Rich fields now use one field for editing & export, & another … for display and printing. This is to help minimize the size of the filled form.” Only the terse editable field is exported to FDF; the bulky rendered twin is regenerable.

The decision that radiates everywhere

This one storage optimisation is the single most consequential choice in the form—and the source of most user pain. Because the editable and display fields are separate, the form needs a manual REFRESH to repaint, a “View Editable Fields” command to reach the codes, tooltips that live only on the editable twin, a click-to-edit choreography that swaps which twin is visible, a warning to work around an Adobe bug that leaves the raw code exposed, and a destructive two-pass rebuild when switching richness. A web rebuild collapses all of it into one value rendered live—deleting five pain points at once.

Three shortcut languages, stacked on one value

What looks like “autocomplete” is really three independent expansion stages that can chain—one keystroke can fire all three:

The most elegant—and most telling—part is the first layer. The 2013 version of the card used real PDF combo boxes (dropdowns). The 2015 rich-text rewrite made those impossible (a rich-text field can’t also be a combo box), so the dropdown was reimplemented out of the materials already at hand. The list of options is stored inside the field’s own tooltip, and the Validate script searches that tooltip for the typed code:

One string, three jobs

The same tooltip text is simultaneously the hover help, the “?” pop-up reference, and the lookup table for code expansion. The help text is the autocomplete data. It’s a beautiful economy—and also a documented UX regression: a real dropdown became a modal “type ?, dismiss the alert, retype the digit, tab out” dance.

The render engine: rich text built by hand

A PDF field has no markup engine, so all formatting is hand-assembled as a richValue span array in JavaScript. The largest single field action (~15.5 KB) is the shared Format script: it walks the value on the ! introducer, maintains styling state, and emits styled spans—suit symbols in the embedded Cards font (hard-capped at 11pt), colours from a palette array, bold/italic/underline, super/subscript, size steps (!- floors at 6pt), tabs, and !U+XXXX; Unicode—then assigns the finished array to the D_ twin’s richValue. A built-in legend function, gFVbR, documents the whole !-language; its colour set is exactly what a rebuild needs:

REFRESH: there is no redraw API

Since writing a value is the only render trigger, the whole-card REFRESH command works by re-writing every required field’s value back onto itself—but with a twist. Readers optimise by skipping the Format pass for fields that haven’t changed, so REFRESH prepends a ^  caret (and sometimes appends a character) precisely so each value looks changed and the optimisation is defeated:

// gFiF case 42 — REFRESH, looping this.numFields
if(fieldName.substr(0,2)==='D_') continue;   // skip display twins
if(!f.required) continue;                    // "required" flag == "is a rich field"
f.value = '^ ' + f.value;                     // force a "change" → re-fires Format

A hack to force a hack—and a perfect illustration of the whole form’s character.

05The command surface

Three command channels, because no single one suffices

An AcroForm cannot add toolbar buttons, ribbon items, or application menus. So the author built the app’s entire chrome out of three things a PDF does offer—each with a different trade-off.

1 · Bookmarks as a menu (127 entries)

The PDF outline (bookmark) pane is the only built-in, always-visible, off-page surface that can carry a clickable JavaScript action. The guide is candid (p.21): “our bookmarks function more like a menu of commands rather than links to other parts of the document.” Each bookmark’s action is a thin dispatch snippet—usually a single call into the shared library—so the menu and the field events reuse the same functions.

The pane is hidden by default, so the guide teaches—repeatedly—how to reveal it (View › Show/Hide › Navigation Panes › Bookmarks) and to click the flag icon, not the label (clicking the name can start a rename). 2 · Action Areas as real PDF layers

Overlaid on the card are pale pink/yellow clickable rectangles—“Action Areas.” Uniquely in the whole form, these are genuine PDF Optional Content Groups (layers), named Layer AA…. Before printing, gFbP calls this.getOCGs() and hides every such layer so they don’t appear on the printed card. But the form is honest about the limits of this: in Adobe Reader “the layers will be hidden but the active print process will still print them,” so it pops a warning offering a reprint—a workaround that admits it only fully works in Foxit.

3 · Field-event hot-zones

The remaining controls are emulated as fields. The classification circle, the “imitation checkboxes” (plain text fields cycled through styles by gFpB and mirrored to a B_ display twin), and dynamically-created V_/Box overlay fields all carry mouse/focus actions wired at runtime via setAction(…). Clicking a display twin fires gRFF, which swaps the display flags and moves focus to the editable twin—the “click to edit” behaviour. Because these are fakes, buggy readers misfire; the form ships a literal “You have touched field … click the side label at least TWICE” alert, and a dedicated For MPE Users bookmark for one reader that needs the label clicked instead of the box.

06Data, persistence, versioning & sync

An offline-first, schema-versioned document database in an AcroForm

There is no server and no external store. The “data model” is simply the set of field values, which is exactly what a saved PDF—and a far smaller exported FDF—persist. The 601 fields fall into three families: card content (editable + D_ twin pairs), settings (the My_* family), and provenance (I_LastSave, A_JN, Date_A).

FDF—and the Adobe Reader export hole

The compact interchange format is FDF (~30 KB versus a multi-megabyte filled PDF). Import is straightforward (this.importAnFDF()). Export is where a platform gap bites hard:

Limitation → workaround

The recommended free reader—Adobe Reader—cannot write FDF files. The workaround, the Make FDF in Log bookmark, hand-assembles a syntactically valid FDF document (%FDF-1.2, a Catalog object, trailer, %%EOF) one console.println at a time and dumps it into the JavaScript console for the user to copy out by hand. It even emits synthetic aaNote_* pseudo-fields narrating how many fields were skipped. The happy path for sharing your card requires opening a developer console.

Versioning & migration—a schema-migration DSL inside a bridge card

The 2020 update renamed many fields, which would have broken FDF interchange with older cards. The fix is a tiny declarative rename language stored as a string, gRtFL:

JumpRaiseMinorOther>JumpRaiseMinor   UnusualNTOther>UnusualNoTrump
Player~.>_   Resp~.O>_O   Over1NTInterfMore>Over1NTInterf   // OLD>NEW (+ templated prefix rules)

At load, gMRFL parses these rules and, to actually receive an old field’s value (since importAnFDF only fills fields that already exist), the Make Retired Fields command mutates the form’s own AcroForm at runtime—addField()-ing hidden, read-only, red-bordered placeholders with the old names, then merging them onto their new partners. It’s a self-modifying import shim, and it self-disables (the gAfW flag) the moment addField throws in a reader without editing rights.

“Partner sync” is cosmetic, not replication

A partnership keeps one system file. The Swap Players command (gFsP) swaps only the two PlayerName_A/B and PlayerNo_A/B slots (carrying their formatting), and Print Partner’s Card is just swap → print → swap-back. There is no second copy, no merge, no conflict resolution—the right mental model is “one document, two views.”

A privacy footnote

On every save, I_LastSave is stamped with the form revision and build (21E29.2115), an exact timestamp, the reader’s type/version/variation, the OS platform, a refresh counter, and the saving user’s identity.name / loginName. A shared card quietly leaks who, what, and where it was last edited—something a rebuild can and should drop.

07Capability detection & graceful degradation

PDF-era progressive enhancement

Rather than assume a reader’s capabilities, the form probes them at open and then degrades along a continuous scale. It is unusually disciplined for a PDF:

  • Rich-text probe. At open, the form writes a two-span richValue to a test field and reads it back. Success sets the rich ceiling gRMx = 8 (full rich); failure drops to 5; Nitro is pinned to 5. So “Richness” isn’t a binary—it’s a scale of 8 (Rich) / 5 (enhanced) / 2 (Basic) / 1 (None), and unsupported readers auto-downgrade so the card still looks reasonable.
  • Viewer fingerprint. Five properties (viewerVersion, formsVersion, viewerType, viewerVariation, platform) drive per-reader branches—Foxit gets four-space tabs instead of \t, no progress thermometer, and the alert-code remap.
  • Self-disabling features. If addField throws, gAfW flips to false and every feature that needs to create fields (display twins, retired-field shim) quietly stands down rather than crashing.
  • Reaching into the host. Acrobat’s “Highlight Form Fields” preference paints over the classification circle’s fill colour, so gFcb directly writes app.runtimeHighlight = false for the session—the form actively toggles a host-application setting to keep its own colours visible.

The guide encodes the residue of all this as a per-reader compatibility matrix (p.45): Adobe Reader does rich text but can’t export FDF; Acrobat Pro/Standard and Foxit 10.1.3+ do both; Apple’s Preview should be avoided because it flattens the form in a way Adobe later misreads.

09Self-test, provenance & candid fragility

A form that is honest about its own precarity

What makes the form genuinely interesting—beyond the engineering—is how openly it acknowledges that it might not work, and builds tooling around that fact.

JavaScript proves it’s alive by disappearing

The big pale-yellow panel on page 1 isn’t decoration—it’s a canary. The field A_JN is shown by default and deleted by working JavaScript on open (a two-pass display.noPrint → display.hidden state machine that even rescales its own text by counting trailing !- markers). If it’s still visible, your JavaScript engine never ran. The panel helpfully lists the exact three Acrobat preferences to enable, then erases itself when those preferences take effect. Clicking the ABF logo brings it back on demand.

Status Check, and a self-aware “I’m dead” state

The top bookmark, gFScK, prints a timestamped report—file size, per-type field counts, the last-save provenance—and crucially detects the fatal state: if(this.numFields==0) it reports “There are no fields in this file - it looks FLATtened/signed … MOST bookmarks will do nothing.” The app can diagnose its own death.

Developer tools shipped to end users

  • gVrbL == 666 is a verbosity backdoor that dumps every member of the document object to the console; higher values unlock progressively deeper internal logging. It is present in the shipped product.
  • The Miscellaneous bookmark group is described in the guide as “most useful for the developer of the form.”
  • At least one genuine bug rides along untouched: gFpB’s error path calls cosole.println (missing the “n”)—dead code that would throw if ever reached.
Why this matters for the rebuild

The candour is the strongest argument for the project. None of this scaffolding—the canary, the restart ceremony, the console-dump export, the retired-field shim, the per-reader click rules—is about bridge. It is all overhead the platform imposes. A web page deletes the overhead and keeps only the card.

10Implications for a web rebuild

Keep the card, collapse the cleverness, drop the overhead

Mapping the analysis onto the project goal—reimplementing “the parts users care about” as a web page—sorts the form’s machinery into three buckets.

VerdictFrom the PDFIn the web rebuild
KEEP The regulated section/field schema (1–10), the Classification enum + Brown-Sticker flag, the player identities, the shortcut-code conveniences and suit symbols, the WBF abbreviation glossary, swap-to-print. A typed field schema; an enum + boolean; live suit/colour rendering with a small code language ported from gFVbR + the gClrA palette; an abbreviation autocomplete.
COLLAPSE The editable/D_ twin split; the three render tiers; REFRESH; the three stacked shortcut layers; settings broadcast via Calculate scripts. One value per field, rendered live (CSS handles colour, overflow, fonts); a single reactive update; one input pipeline; a plain settings object.
DROP The 127-bookmark menu, the canary field, capability probes, per-reader click rules, the retired-field shim, the console-dump FDF export, the identity/viewer provenance leakage, signing/flatten hazards. Normal UI controls; nothing to probe; a JSON document with a schema-version field and a migration table seeded directly from gRtFL; no surveillance metadata.

A clean data model

The web version needs only: a flat map of the editable card fields (the D_ twins are regenerable); a typed settings object (colours, fonts, checkbox style, date format, custom codes); player identity as a two-element array with a “primary” pointer instead of physically swapping values; classification as enumerated metadata; and a small provenance block (schema version + timestamp) in place of I_LastSave. Versioning becomes a one-time migration on load, driven by the rules already written down in gRtFL—no self-modifying fields required.

The opportunity

Everything the form fights for—rich coloured text, live rendering, a real menu, overflow you can see, painless sharing, working dropdowns, no setup ceremony—is free in a browser. The hard part of the rebuild isn’t the technology; it’s faithfully reproducing the regulated bridge content and its semantics. This document is the map of which parts of the original are worth carrying across, and which were only ever there to satisfy the PDF.

＊How this was analysed

From a sealed binary to a readable map

The form ships as a compressed binary; none of the above is visible by opening it. The analysis decompressed all 570 FlateDecode streams in ABF_Card_FORM.pdf, recovered the 183 /JS action blocks (unescaping the PDF string literals, including the \+EOL line-continuation packing), beautified the function library, and extracted the 601 field names, 690 tooltips, and 127 bookmark titles. The 53-page Usage Guide was read both as text and as rendered pages. A fan-out of analysis agents then examined seven facets in parallel—the JS engine, the rich-field system, the bookmark menu, the user workflow, platform limits, the bridge domain, and the data model—each verified against the source and cross-checked by a completeness pass that corrected several first-draft errors (for example, that gFsH is the help displayer rather than a show/hide helper, and that PDF page 1 is the Outside page). Extraction scripts and intermediate artifacts live alongside this file under abf/extracted/.
