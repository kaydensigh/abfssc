System Card · Web Rebuild · Design

Rebuilding the ABF System Card as a static web app

A single typed document, two presentation paths: a responsive on-screen card for phone and desktop, and a regulation-faithful PDF produced by filling the original blank form. This is the implementation design — grounded in the reverse-engineering work and fact-checked against the real PDF.

React 19 + Vite + TS 100% static pdf-lib read + fill mobile + desktop ~295 logical fields full !-code markup
♠ ♥ ♦ ♣

01Overview & the directions we took

Keep the card, collapse the cleverness, fill the original for print

The companion analysis (architecture.html) showed that the existing PDF packed what is essentially an app inside a PDF. The web platform provides most of those features for free. The design goal, then, is narrow: faithfully reproduce the regulated bridge content and its semantics, and let the platform do the rest. The build target is a static site deployable to GitHub Pages / Cloudflare Pages.

Several genuine forks were settled before this design. Each is recorded here with the alternatives considered, so the “why” survives.

DecisionOptions consideredChosen & why
Product scope Digital-only · print-faithful only · both Both, equally. One data model feeds a responsive screen UI and a regulation PDF. No compromise on either; the model is the single source of truth.
Print output Re-draw the regulated layout ourselves (pdfmake/jsPDF) · fill the original blank PDF · hybrid Fill the original blank. We ship ABF_Card_FORM.pdf and write values into its existing AcroForm fields. The output is the regulation form — conformance by construction, zero layout to maintain, and it re-opens in the legacy ecosystem.
Round-trip Re-openable by us only · also by the old form/tools · both Both, for free. Because the export is a filled ABF form, the legacy ecosystem can read it and so can we. An optional embedded marker gives a lossless fast-path on re-open.
Build stack Vanilla TS · Svelte/Solid · React React 19. Most widely used; the form→sections→grid structure maps cleanly onto components. All versions verified live (see §08).
Markup scope Suits + colour only · + sub/super · full language Full gFVbR language. Suits, the 10-colour palette, b/i/u, super/sub, size steps, tabs, Unicode escapes, lock/force, grave shortcuts — minus internal diagnostic injectors. One render engine, used twice.
Save & share local autosave · QR · share-link · JSON file (any combination) All of them, with autosave as the always-on base. Local-only storage is clearly communicated. Share-link is designed-in but its URL-size ceiling is verified after MVP.
Legacy import none · filled PDF · FDF · all Filled PDF + FDF + our own files. Entirely client-side, normalized through the form’s own gRtFL rename rules so even pre-2020 cards map correctly.
The one rule everything hangs off

A single typed Card object is the only source of truth. The screen, the PDF, the JSON file, the share-link and the QR code are all projections of it. The rich rendering (suits, colour, formatting) is derived, never stored — similary to the “editable value + regenerable display twin” the original form used.

02System shape

One model, three projections, two foreign formats
┌──────────────── Card (typed model) ────────────────┐ │ ~295 logical fields · responses[opening][bid] grid │ │ 10 booleans · classification enum + brownSticker │ │ players[2]+primary · settings · {schemaVersion, ts} │ └───────────────┬─────────────────────────────────────┘ zustand store + idb (IndexedDB) autosave, debounced ── LOCAL ONLY │ ┌──────────────┬─────────────┼───────────────┬──────────────────┐ ▼ ▼ ▼ ▼ ▼ SCREEN EXPORT pdf EXPORT json SHARE link/QR IMPORT React form fill blank canonical fflate → #frag legacy PDF / FDF live render ABF_Card_FORM interchange + QR (qrcode) / our json / our pdf §8 grid /V + D_ /RV+/AP pdf-lib read → gRtFL (table↔stack) (renders w/o JS) rename → Card

Three layers do the work. They are deliberately decoupled so the pure logic is trivially testable and the foreign-format mess is quarantined.

Puremodel/ + render/no React, no DOM, no pdf-lib

The Card type and the !-code render engine. A single pure function renderCoded(raw) → Span[] is consumed twice: by a React renderer for the screen, and by a PDF appearance-builder for export. Build it once; fidelity everywhere.

▼
Edgesio/ + state/all foreign formats live here

pdf-lib is touched in exactly one directory behind a two-function adapter. FDF parsing, share-link encode/decode, JSON, and the zustand store + IndexedDB persistence sit beside it. Card identity / import-resolution logic lives in state/identity.ts.

▼
Shellui/the only place React lives

One component per regulated section + the masthead, the §8 grid, the six field-kind components, the storage indicator, and the import-resolution dialog. The editing DOM is never made to print — printing means “export the official PDF.”

03Data model & card identity

A flat map of logical values, a ragged grid, and three orthogonal stamps

The persisted Card is a flat keyed map of the ~295 editable logical fields, the §8 grid as a nested partial map, ten booleans, a classification enum plus an independent brown-sticker boolean, a two-player array with a “primary” pointer, a typed settings object, and a two-field provenance block. Every D_/B_/V_/Z_/My_/I_ machinery field of the original is dropped — they are regenerable PDF plumbing.

type CodedText = string;   // the raw !-code source the user typed; rendered form is DERIVED, never stored
type SuitOpening = '1C'|'1D'|'1H'|'1S'|'1NT'|'2C'|'2D'|'2H'|'2S'|'2NT';

interface Card {
  id: string;            // stable UUID v4 — plan identity; never changes through edits/round-trips
  schemaVersion: number; // shape of our model (gate for migration)
  revision: { label: CodedText; counter: number; parent?: number }; // which version
  provenance: { schemaVersion: number; savedAt: string };  // ISO; NO identity/viewer/OS leak

  players: [Player, Player];   // slots A=0 / B=1
  primaryPlayer: 0 | 1;       // swap-to-print = a pointer flip, no data movement
  basicSystem: CodedText;      // coded (1 Standard / 2 2-over-1 / 3 Acol / 4 Precision / …)

  classification: 'green'|'blue'|'red'|'yellow'|'unset';
  flags: CardFlags;            // the 10 boolean agreements (see below)

  s1_openings: Section1; s2_preAlerts: Section2; /* … */ s10_notes: Section10;
  s8_responses: ResponsesGrid; // nested PARTIAL map (the grid is ragged)

  settings: CardSettings;      // display prefs, separate from card content
}

// The §8 grid is RAGGED — only the cells that exist for a given opening are present.
type ResponsesGrid = Partial<Record<SuitOpening, Partial<Record<ResponseBid, CodedText>>>>;
Verified correction · the §8 grid is hierarchical, not flat

Probing the real PDF shows the grid cells are hierarchical AcroForm fields: a parent field Resp1C with child partial names _1D, _1H, … Other — so the fully-qualified name is Resp1C._1D (dot separator), not Resp1C_1H. Parse the field tree (parent /T + child /T), or split fully-qualified names on the first dot. And beware: Resp1NT2D, Resp1NTDoubled, Resp1NT2CStyle are separate top-level coded fields (they carry their own tooltip code-lists), not grid cells — the real 1NT cells are Resp1NT._3C … Resp1NT._4S/Resp1NT.Other. Conflating these mis-routes data on import and export.

The ten “checkboxes” are booleans to us — but text fields in the PDF

On our side these are unambiguously booleans:

interface CardFlags {
  IsBlackwood: boolean; IsGerber: boolean; IsAskingBids: boolean;
  IsCueBids: boolean; IsNTCheckback: boolean; IsCanape: boolean; IsBrownSticker: boolean;
  Is4thForcing1Round: boolean; Is4thForcingGame: boolean;  // 4th-suit-forcing is two agreements
  OneNTMayHave5Major: boolean;
}
Verified correction · do not call the checkbox API

Probing by /FT confirms IsBlackwood and the rest are /Tx text fields (“imitation checkboxes” the original cycles through tick styles), not /Btn checkboxes. pdf-lib’s form.getCheckBox(name) will throw. On export, write the tick-style glyph value (My_CheckBoxValues: Big-X / Small-Black / Tick / Cross) into the text field and its B_ twin; on import, read a non-empty/“on” value as true. Classification is likewise a text field, and the blank stores the literal sentinel not set (not Green/G/1) — settle the colour literals from gFpB in the JS before the import/export mapper is built. The only true non-text fields are the two print-mirror radio groups Z_MyClass/Z_Sticker (already dropped).

Identity, revision and schema are three different things

Conflating them is the classic bug, so they are kept orthogonal:

  • id (UUID) — plan identity. Minted once; survives every edit, export, re-import and player swap. This is what tells “a different pair’s card” from “another revision of our card.”
  • revision — which version. A monotonic counter + the user-facing label. Replaces the original’s I_LastSave ordering without its viewer/OS/user payload.
  • schemaVersion — which shape of our model. A migration gate. Bumping the app schema must not look like the user edited the card.

Migration on load is a one-time pure transform, never self-modifying fields. Two distinct layers, run in order: first a legacy field-name rename (current AcroForm names ← pre-2020 names) seeded from the form’s own gRtFL DSL; then, only when a second schema eventually ships, a schemaVersion ladder. For v1 there is exactly one schema, so the ladder is a single guard, not infrastructure (see §10 YAGNI).

Import resolution — the “new file vs. partner’s new version” problem

After rename + migration produce a candidate Card, branch on id. Unknown id (every legacy file, since it has no UUID of ours — mint one) → new card. Known id → it’s our plan again — a new revision or a partner’s edits → offer replace / keep-both / compare, using revision.counter/savedAt to label which is newer. To distinguish linear succession (their rev 6 descends from my rev 5 — safe replace) from a fork (both forked from rev 5 — must compare), carry a revision.parent pointer; absent that, route all same-or-lower counters to compare.

What is dropped — and a stale-artifact caution

Verified correction · build the field registry from the asset, not the text dump

The live ABF_Card_FORM.pdf has 690 AcroForm fields (688 text + 2 radio), not the 601 the extracted field_names.txt lists — the text dump is stale relative to the asset we actually ship. Build the canonical field registry by enumerating the live form’s getFields(), then drop the machinery prefixes (D_=245, My_=13, Box=10, B_=9, A_=7, I_=6, Q_=5, T_=5, V_=4, Z_=2). The bare _1D…_4S “column header” fields the drafts wanted to drop don’t exist as standalone fields — they’re only the child partials of the Resp parents; the headers are pure render-time labels. The “~295 logical” figure counts grid cells collapsed into the matrix; the flat getFields() non-machinery count is ~383. Pin the exact registry once, in code, against the asset.

04The render engine

One pure function, three stacked code layers, two consumers

~295 logical fields and all 52 code-lists are authored in the form’s terse code language, so we must port it. The engine (from clean_2_block20.js) is a string interpreter; our port is a single pure function with no globals, no DOM, no PDF — renderCoded(raw, opts) → Span[] — feeding both the screen and the PDF export.

Three layers, run as sequential passes

LayerIntroducerWhat it does
A · Tooltip code-listdigit / x / 9 / ?Per-field. Expands a code to its phrase (the 52 lists). On the web this becomes the field’s autocomplete; selecting inserts the phrase, which may itself contain !-codes that then render.
B · Grave shortcut` backtickGlobal hash gCdH (seeded by gFrC). Values are themselves !-code strings, so B expands into C. Extensible via user settings (My_Codes).
C · Rich codes!The render engine proper — suits, colour, b/i/u, super/sub, size, tabs, Unicode, lock/force.

The !0–!9 palette is taken verbatim from the source gClrA array (every value confirmed against the architecture doc):

!0 black
!1 red
!2 blue
!3 fuchsia
!4 green
!5 purple
!6 cerulean
!7 brown
!8 dk brown
!9 orange

The full code set to implement: suits !S!H!D!C (and lowercase) — rendered on screen as Unicode ♠♥♦♣ with CSS colour (the embedded Cards font is not needed in the browser; defaults are black spades/clubs, red hearts/diamonds, matching gScxA); colour !0–!9; b/i/u !b!i!u; super/sub !^!v; size !+/!=/!- (suits capped at 11pt, size floored at 6pt); tabs !:/!;; Unicode !U+XXXX;; lock/force !N!= (swallowed by the renderer); save/restore !{ !}; literal !!; alignment !< !| !>. Grave defaults include `1`→1ˢᵗ … `5`→5ᵗʰ, `@`→½, `_`→en-dash, `a`–`e`→α–ε.

YAGNI & verified simplifications

Collapse the original’s dual rich/basic sub-parsers into one rich parser — the “basic” path only existed for low-capability Acrobat readers we don’t have. Drop the diagnostic injectors !$ !* !@ !~ !& (internal field-name/date/build dumps) and the gEit italic fallback (browsers always have italics). The engine must be total: an unknown !x emits the literal !x, so malformed legacy values never throw.

interface Span { text: string; suit?: 'S'|'H'|'D'|'C';
  color: string; bold: boolean; italic: boolean; underline: boolean;
  vshift: 0|1|-1; sizePt: number; align?: 'left'|'center'|'right'; }

function renderCoded(raw: string, opts: {
  fieldDefaults?: { sizePt: number };
  codeList?: Record<string,string>;   // Layer-A list for THIS field
  graveTable?: Record<string,string>;  // Layer-B gCdH
}): Span[];

Consumer (a) — React: map each Span to a styled element (<sup>/<sub> for vshift, CSS for the rest), suits as Unicode glyphs pinned to text-presentation. Consumer (b) — PDF export: the same Span[] becomes the D_ twin’s rich appearance (see §06). The intermediate Span[] is the shared contract, and the boundary the test table asserts against.

Editing UX

Default display is rendered. On focus a cell shows its raw codes in a monospace input with a live preview beside it (debounced; renderCoded is pure and cheap). A markup toolbar inserts the common codes (♠♥♦♣, the colour swatches, B/I/U, super/sub, size) at the caret, so casual users never type a code and power users still can. Coded fields add an autocomplete seeded from the field’s list — strictly better than the original’s modal “type ?, dismiss, retype” dance, same data.

05Import pipeline

Legacy PDF, FDF, and our own files — all converging on one Card

Import is read-only and entirely client-side. Four sources in two tiers: a fast path for files we wrote (our JSON, our PDF’s embedded marker), and a legacy path for filled ABF PDFs and FDF. All converge through one pipeline:

bytes ─▶ load ─▶ read fields ─▶ normalizeName (gRtFL) ─▶ drop machinery ─▶ mapToLogical ─▶ Card │ build responses[opening][bid] from the field TREE

Legacy filled PDF

PDFDocument.load(bytes, {ignoreEncryption:true}) → getForm().getFields(). If the list is empty, the file was flattened or signed (numFields==0) — an accepted limitation: print-only, not importable, surfaced with a clear message rather than an empty card.

Verified correction · two import gotchas

(1) Values are already expanded text. The original’s Validate script replaces a typed code (<3 chars) with its phrase at commit time, so a saved card stores “12-14 Balanced”, not “1”. Import therefore passes values through (they may still carry !-codes, which the render engine resolves) and must not try to reverse-map a phrase to a code. The elaborate code-vs-prose disambiguation is unnecessary for the common case. (2) Don’t call getText() on rich fields. It throws RichTextFieldReadError when a RichText-flagged field has no /V — and all 245 D_ twins are rich. Dropping D_ avoids this, but a naïve “loop every field calling getText()” will throw; guard with isRichFormatted() / try-catch.

Build the §8 grid from the field tree, not from string-splitting flat names (see the §03 correction): walk parent Resp<opening> → child _<bid>/Other. The bare opening field (Resp1C with no child) is the per-row “general approach” cell — a distinct logical field, not a grid cell.

The gRtFL rename normalization

The form’s own migration DSL (transcribed exactly from _openaction_clean.js; the architecture doc’s 5-rule snippet is abbreviated — use the full 8):

Player~.>_   Resp~.O>_O   Resp!ons!Oth!._~._>_   Resp1NT2COther>Resp1NT2CStyle
JumpRaiseMinorOther>JumpRaiseMinor   JumpRaiseMajorOther>JumpRaiseMajor
UnusualNTOther>UnusualNoTrump   Over1NTInterfMore>Over1NTInterf
Verified · the 3 templated rules need algorithmic expansion

The five plain OLD>NEW rules are a trivial static dict. The three ~ rules (Player~.>_, Resp~.O>_O, Resp!ons!Oth!._~._>_) are not name pairs — they are prefix + substring-substitution with !-delimited ignore-tokens, exactly as the source gMRFL parser applies them. You cannot precompute a static map from the DSL alone; port gMRFL’s loop and run it against the live field-name list at build time to materialize the concrete old→new pairs. Several rules are merges (e.g. JumpRaiseMinorOther → JumpRaiseMinor): concatenate rather than overwrite, and log any legacy name that survives normalization yet fails to map.

FDF

Verified correction · FDF needs a Kids-tree walker, not a flat regex

Hierarchical fields (the entire §8 grid) appear in FDF as nested /Kids dictionaries — <</T(Resp1C)/Kids[<</T(_1D)/V(…)>> …]>> — not flat /T(Resp1C._1D)/V(…) pairs. A flat /T(…)/V(…) regex misses every grid cell (the parent has /T but no /V; children lose the parent name). Walk the dict structure, tracking the parent /T while descending Kids. Also handle hex-string names (/T<48…>) and /V-before-/T ordering. The original’s console-dump whitespace tokens (<<R>> etc.) only matter for that hack, but a cheap reverse pass lets us also ingest a file saved straight from the console.

Our own files (fast path)

Our JSON is the canonical Card serialization — no rename, no family-dropping, just a schemaVersion check and the identity branch. Our exported PDF carries an embedded marker (UUID + schemaVersion + revision) so re-open is lossless and instant; absent the marker we fall back to field-by-field reading. Always try the marker first.

06Export pipeline

Fill the original blank — and the one hard spike that makes it print

Everything leaves through four exporters. Three are trivial serializations of the in-memory Card (JSON, share-link, QR). The fourth — the regulation PDF — is the only hard one, and the only one with a genuine decision point.

1 · “Export official card” — fill the blank

We ship abf/ABF_Card_FORM.pdf (535 KB, blank, fields + embedded JS intact) as a static asset, fetch it at export time, and write our values into its existing fields by name (mapping our logical keys back through the same gRtFL table). The output is the regulated form — it re-opens in Adobe/Foxit and in our own importer. We never reproduce the layout.

Field classExample/F flagPrints?
Editable code fieldOpen1C, PlayerName_A32 = NoViewNo
D_ display twin (×245)D_Open1C4 = Print, RichTextYes
The decision point · what prints is the D_ twin, not the field we “own”

Verified against the asset: editable fields are NoView (never print); only the D_ twins are Print. So if we fill only the editable fields, the printout is blank until a recipient opens the file in a JS-capable reader and runs the form’s REFRESH. That fails on every phone viewer, Preview and browser PDF view — unacceptable for “regulation-faithful output.” Strategy A (chosen): we pre-populate the D_ twins’ rich appearance ourselves, reusing the render engine. Strategy B (fill editable, rely on the reader’s JS) is rejected for output.

Verified feasible · Strategy A works, end-to-end

pdf-lib has no rich-text API (setText writes plain /V; enableRichFormatting only flips a flag). But the spike confirmed the real path works: (1) set /RV by hand on the field dict (for Acrobat re-editability — the D_ twins already carry exactly this structure), and (2) build the visible appearance yourself via field.updateAppearances(font, provider), where provider returns a raw operator array — setFillingColor, setFontAndSize('Cards', 11), showText(cardsFont.encodeText('S')), Ts rise for super/sub. This saved a valid PDF with the embedded suit font as the AP resource. The Cards font is free: it’s already embedded in the blank’s /DR (a 104-glyph TrueType containing all four suits), so an appearance referencing /Cards needs no new embed. Keep NeedAppearances=false so viewers honour our appearance verbatim. This is the #1 spike for M1.

Provenance is dropped for free: we never write I_LastSave/A_JN, so our export leaks no name, OS, reader or login — a real privacy win over the original, at zero cost. The only stamp is the model’s {schemaVersion, savedAt}.

2 · JSON, share-link, QR

JSON is the compact, lossless, diffable interchange (store the raw !-code source per field; round-trip JSON→JSON must be identity). Share-link: JSON → fflate raw-deflate → base64url → URL fragment (#, never sent to a server). QR is generated from that same URL.

CardJSONdeflatebase64urlURL
Typical / sparse~1.9 KB~0.5 KB~660~700
Heavy (full grid + many fields)~15 KB~1.3 KB~1,698~1,722
Pathological (long pasted notes)——~1,900~1,931

Compression is ~10:1 (repeated codes + stock phrases). Even a heavy card sits under the conservative ~2 KB URL ceiling, so share-link is viable for the overwhelming majority of cards — but free-text notes are the failure mode. Set the fallback trigger at ~1,800 chars (degrade to JSON-file / QR-of-a-short-card, never truncate), and verify against real payloads after MVP. QR has a tighter real ceiling (~800 chars to stay scannable on paper), so offer QR only below that bound.

Embedded marker caveat

The lossless re-open marker needs a home that survives. Z_AppState doesn’t exist in the form — we’d addField it, which works in our output (pdf-lib writes it directly, no reader-JS needed) but isn’t “guaranteed” through third-party re-saves. Store the marker JSON in both a custom XMP/Info key and a hidden text field, and treat absence gracefully (fall back to field-by-field import).

07UI & the hard flows

Responsive sections, a grid that restacks, and three flows that bite

One route per regulated section plus the masthead — matching the regulated schema 1:1 makes the card legible to anyone who knows the paper form. Six field-kind components (RichTextField, CodedField, Checkbox, ClassificationSwatch, PlayerField, NotesArea) cover everything.

Responsive layout

  • Section nav: tab strip on desktop, drawer + section <select> + thumb stepper on mobile. Present sections in natural order 1→10 — the PDF’s right-column front cover is a print constraint and must not leak into editing.
  • §8 responses grid: a real <table> with sticky header row + first column on desktop; a stacked accordion, one panel per opening, on mobile. Both read/write the identical responses[opening][bid] cells, so a viewport change never reshapes data.
  • Classification: four colour swatches as a radio group; clicking the active one clears to unset (the original’s radio-from-one-string behaviour). Brown Sticker is a separate checkbox beside it. A helper line links out to current ABF regs for the colour semantics (deliberately out of file).
  • §5 play conventions: render the natural _NT/_S pairing as two columns (vs-NT / vs-suit), stacked on mobile.
Flow 1 · “Autosave is on this device only”

The biggest UX risk is a user assuming autosave = cloud sync. Three coordinated affordances: a persistent storage indicator (“Saved on this device · time”, a device icon, never a cloud) whose explainer names the exact loss vectors (clear site data / other device / private browsing); a prominent Back up / Export button framed as “the real save”; and failure-prevention copy at export moments (“that file, not this browser, is your safe copy”). Request navigator.storage.persist() and escalate the indicator to amber if denied — Safari/iOS is the known gap.

Flow 2 · Import resolution (new file vs. partner’s new version)

On any open, parse → normalize → read {id, revision, schemaVersion} → branch. Dialog A (unknown id): “This looks like a different card …” → Open / Keep both / Cancel (no destructive default). Dialog B (matching id — the partner hand-off): “This is an update to Smith / Jones. Theirs is revision 7; yours is 5.” → Replace mine (old kept in history) / Keep both / Compare → a field-level diff with per-field keep-mine / take-theirs. This is the one place we add reconciliation the original lacked, because two people now genuinely edit separate copies. Use the revision.parent stamp to route true forks to Compare rather than auto-picking a “winner.”

Flow 3 · Swap-to-print & Print

Players are [A,B] + a primaryPlayer pointer; “Swap players” = flip the pointer (instant, reversible, zero data risk). “Print partner’s card” = flip → export → flip back. The Print button simply runs “Export official card,” then hands the PDF to the browser/Acrobat print path — there is no separate print stylesheet. Open question worth an early spike: whether browsers can be driven to print an in-memory filled PDF directly, or the realistic flow is “download, then print from a viewer.” And the print promise rides entirely on the Strategy-A spike (§06) — surface a graceful degradation path if it slips.

Accessibility: each section is a labelled landmark; the §8 grid is a real table with scoped headers and arrow-key cell navigation; classification swatches carry text labels and aria-checked (never colour alone); the live !-preview is aria-hidden with the raw code field carrying the accessible value.

The single biggest technical risk · pdf-lib maintenance

pdf-lib is the only library that can both read and fill AcroForm fields client-side, and it’s low-activity. Mitigation, layered: (1) pin 1.17.1 + lockfile; (2) hide it behind a thin adapter — all pdf-lib calls live in io/pdf/adapter.ts behind readFields() and fillForm(), so a swap is one directory; (3) keep pdf.js as a read-only import fallback behind the same interface (it can’t fill, so export stays on pdf-lib).

09Testing

Four families, one contract test that protects the whole product
  1. Render-engine unit tests (table-driven). Pure and central — test it hardest. A table of !-code / grave inputs → expected Span[], asserted at the tokens boundary so both consumers are covered at once. Every suit, every palette colour, b/i/u, super/sub, size clamps, tabs, !U+, the grave defaults, and code-list expansion (a digit → phrase whose phrase itself contains !H). Include malformed inputs to lock in graceful degradation.
  2. The IO round-trip contract test (highest value). Fixture = a known filled legacy ABF PDF. Assert import(pdf) → Card₁, export(Card₁) → pdf', import(pdf') → Card₂, and Card₁ deep-equals Card₂ (ignoring the provenance timestamp/revision). This proves the adapter, the gRtFL normalization, the grid tree-mapping, and the render→D_-appearance emission all compose losslessly. Add an FDF fixture too. Build it early — it’s the harness that turns the export VERIFYs into CI guarantees.
  3. Schema-migration / rename tests. Drive the gRtFL table with the real rules — the five simple renames plus the three algorithmically-expanded templated rules against the live field set — and a “known UUID, newer revision” case for the resolution branch.
  4. Grid component tests. The §8 matrix renders the right (ragged) cells; editing writes to responses[opening][bid]; the coded combo seeds from the lists yet accepts free text; classification is one enum with an independent brown-sticker boolean.

Fold the asset-probe scripts (field flags, font coverage, share-link sizing) into abf/ tooling as a CI guard against asset drift — they are what proved the facts in this doc.

＊How this design was produced

Drafted from the artifacts, then fact-checked against the real PDF

Six section authors drafted this design directly against the extracted artifacts (architecture.html, the seven research reports, the field/codelist dumps, and the beautified JavaScript). Three adversarial verifiers then fact-checked the riskiest claims — several by probing the live ABF_Card_FORM.pdf with pdf-lib — which corrected real errors before they reached this page: the §8 grid is hierarchical (not flat); the “checkboxes” and Classification are text fields; editable fields store expanded text; the live form has 690 fields (the text dump is stale); pdf-lib has no rich-text API but Strategy A is nonetheless feasible; and the blank is 535 KB with the suit font already embedded. The remaining open items are collected honestly in §10 rather than smoothed over.

ABF System Card — web rebuild, implementation design. Companion to abf/architecture.html (how the original works) and card-content-model.md (what the card captures). Target: a static React app reproducing the regulated bridge content; the regulation PDF is produced by filling the original ABF_Card_FORM.pdf, never re-laid-out. Suits, naturally: ♠♣ ♥♦.
