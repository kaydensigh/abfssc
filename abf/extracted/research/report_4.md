# DIMENSION 4: PDF/JS platform limits & workarounds

## Summary
The ABF card is a full application smuggled into a 2-page Acrobat AcroForm, so it inherits every constraint of the Acrobat JavaScript host: no custom toolbar/menus, no DOM/HTML, no uniform cross-reader API, fixed-size fields that silently truncate, an incremental-save model that bloats files, and the ever-present risk that JS simply never runs. The authors (Rex Livingston = "RoL") built a layered defense almost entirely out of workarounds: a two-pass self-erasing yellow "A_JN" warning field that proves JS is alive, active manipulation of app.runtimeHighlight plus a five-property viewer fingerprint for feature-detection, paired editable/D_ display fields to shrink saved state, ~127 bookmarks repurposed as the command menu, forced Save-As discipline, file-size self-checks with concrete byte thresholds, and an exportAsFDFStr-to-console trick to give Adobe Reader an FDF export it otherwise lacks. Nearly every "feature" in the guide is really a workaround for a platform limit, and most leave a residual user cost (extra clicks, Save-As hygiene, per-reader click gestures, manual Preferences setup, reader-specific feature gaps).

## Report
# PDF/JS Platform Limits & Workarounds (verified & deepened)

The fundamental constraint: a PDF form can only execute Acrobat's JavaScript API (`this.getField`, `app.alert`, `display.*`, `console.println`, `this.exportAsFDFStr`, document event hooks WillPrint/WillSave/WillClose), only when a JS-capable reader is present AND the user has trusted/enabled scripting. There is no HTML/CSS/DOM, no custom toolbar buttons, no persistent app state beyond field values, and behavior diverges sharply between readers. The whole form is scaffolding around these facts. I re-read the form PDF page 1 (the yellow box), `clean_0_block36.js`, `clean_1_block37.js`, `js_all_clean.js`, and the guide text; several draft claims were wrong and are corrected below.

## 1. "Is JavaScript even running?" â the two-pass self-erasing field (CORRECTED)
The verbatim overlay text on page 1 is *"If you are downloading this form, you need to save it to your local disk and open it locally. ALERTS: 1. Can you see the bookmarks? 2. This form needs javaScript. ... When opening this form, this field should vanish if javascript is active. _____Click in this field should hide this message_____ IF....that leaves a scroll bar or other residue ... THEN....click once or twice anywhere in the card should clear it."* (draft mis-quoted the order/wording).

The real BLOCK 0 logic is **not** a single noPrint-then-hidden; it is a guarded two-pass toggle that also rescales the text:
```
if(this.numFields>0){var cAXjwf=this.getField('A_JN');if(cAXjwf!=null){
 if(cAXjwf.display!=display.noPrint){ var cAXjwv=cAXjwf.valueAsString;
   if(cAXjwv.length>300){ cAXjwf.display=display.noPrint; ... // count trailing '!-' markers, set textSize=14-n
 }}else{ cAXjwf.display=display.hidden; }}}
```
First pass (field still visible) sets `display.noPrint` and adjusts `textSize`; only a *later* invocation (display already noPrint) sets `display.hidden`. The field is the canary â it disappears only if the script fired. `numFields==0` (flattened/signed) short-circuits everything. Manual fallback: clicking the field also hides it (overlay text *"Click in this field should hide this message"*), and it can be re-summoned via bookmarks **"The JavaScript Warning"** / **"Manage Java Warning"**.

## 2. runtimeHighlight is actively WRITTEN, not just read (CORRECTED â draft's biggest error)
BLOCK 1 is `gFcb(4);var fAXqux=true;if(typeof(app.runtimeHighlight)==='boolean')fAXqux=app.runtimeHighlight;if(fAXqux)gFsH('Cls-H');`. The draft claimed `gFsH('Cls-H')` does "classification-highlight handling." Wrong on two counts:
- **`gFcb(4)`** *sets* `app.runtimeHighlight` â `gFcb(lBXRoLA)`: with arg 4, if the reader exposes the boolean it forces highlight **off** so the classification circle fill-colours become visible (`if(!app.runtimeHighlight)lBXnfx=true;` then `app.runtimeHighlight=lBXnfx;`), and because `lBXRoLA<4` is false it suppresses the explanatory alert. This is the real workaround: Adobe's "Highlight Form Fields" preference masks the BLUE/colour fills the card uses for system classification.
- **`gFsH` is the HELP-message displayer**, defined once in block 37: `function gFsH(qDXRoL){... qDXm=gHlpH[qDXxv]; ... gFaA(0,3,qDXttl,qDXm);}`. It looks a tag up in the `gHlpH` help dictionary, substitutes markup (`<n>`âCR, `<T>`âtab, `<L>`/`<R>`âparens), and pops an alert. So `gFsH('Cls-H')` shows the "Classification highlight" help/warning when the reader couldn't disable highlighting. There is exactly ONE `gFsH` definition â the draft's "name collision/overload" theory is FALSE. ("gFsH" = show-Help; the orchestrator's "show/hide" gloss is itself imprecise, though the 64 call-sites count is correct.)

The help text itself lives inside form fields as `<H>tag:text<H>` runs, parsed lazily into `gHlpH` by `gFbH` (`Rebuild.HelpHash`) â a workaround so help content can be patched via FDF.

## 3. Per-reader feature-detection & fingerprinting
`clean_0_block36.js:172` builds the diagnostic `'ViewerInfo: _v:'+app.viewerVersion+' f:'+app.formsVersion+' t:'+app.viewerType+' V:'+app.viewerVariation+' o:'+app.platform`. `typeof gFxWP=='function'` guards check the print library loaded (BLOCK 181). `console.println` is the only log, visible solely via the **"Expose Java Console"** bookmark. Document event hooks are named `gFxWP`(WillPrintâ`gFbP(0)`), `gFxDP`(DidPrint), `gFxWC`(WillClose), `gFxWS`(WillSave) â the form's lifecycle is wired to Acrobat document actions because that is the only event surface available.

## 4. Reader-bug click workarounds (DEEPENED)
BLOCK 3 is a mouse-down handler (`console.println('MD '+event.targetName)`) that alerts *"You have touched field ... This is a problem with your PDF Reader program ... WORKAROUND: We expect you will need to click on the side label at least TWICE."* Imitation checkboxes are plain text fields faking checkbox behavior, so buggy readers misfire. Guide p.24/p.30 add the per-reader rule: click anywhere "between the start of the side heading and the right edge of the box," **except** *"Master PDF Editor is different: please click on the side heading label and not in the box"* â there's even a dedicated bookmark **"For MPE Users"**.

## 5. The "Unchanged Text Field" exposure bug (NEW â draft missed it)
Guide p.27: PDF readers skip the formatting pass for fields entered but not changed, which leaves the *editable* twin exposed instead of the `D_` display twin. *"This works in Foxit Reader but not in Adobe Reader. From revision 21E09, in Adobe Reader we provide a message ... if any editable fields are still exposed when you print or save ... we will provide a warning about the need to first REFRESH."* This is a concrete reader-specific workaround tied directly to the paired-field design.

## 6. No toolbar â bookmarks + Action Areas are the chrome
~127 bookmarks act as the command palette. Beyond the draft's list, verified titles include **"Make FDF in Log"**, **"Shrink file"**, **"File SIZE"**, **"Font removal"**, **"Text Overflow"**, **"Signing Hints"**, **"For Foxit/Nitro/Adobe/MAC/MPE Users"**, **"The Display Fields"**, **"The Editable Fields"**, **"WARNING: Pasting text"**, **"Refresh at Open"**. Guide p.21: *"our bookmarks function more like a menu of commands rather than links ... click on the action box/flag rather than the name."* On-page pink/yellow "Action Areas" handle REFRESH/Swap/Show-Priorities.

## 7. Save bloat & file-size self-check (PRECISE THRESHOLDS â NEW)
Forced Save-As (guide p.15: *"Why does the simple SAVE ... keep failing and asking for a new name?"*) defeats Adobe Reader's incremental-update bloat (>5MB). The open-time and Status-Check size test uses real byte constants: the soft "Reminder, the form is getting large" fires at `kMXzL=((gSPNs==2)?1810000:1550000)` (~1.55â1.81 MB) and a stronger warning at `xEXzL=((gSPNs==2)?2100000:1700000)` or `5500000` depending on context â i.e. ~1.7 MB, 2.1 MB, or 5.5 MB. The threshold is computed (depends on `gSPNs`, the sticker-page state) and the guide confirms it's configurable via the Advanced Guide. The paired editable/`D_` architecture (2020 redesign, "one field for editing & export & printing ... to help minimize the size of the filled form") is itself a size workaround: editable fields hold terse codes, `D_` fields hold rendered text, and `gFmkR` lazily `addField('D_'+name,'text',...)` creates the display twins.

## 8. FDF export â and the exportAsFDFStr-to-console trick (RESOLVED open question)
FDF (~30 kB vs the multi-MB PDF) is the compact transport, but *"Sadly, this is not available in any version of Adobe Reader"* (p.31) because `this.exportAsFDF` (file write) is blocked in Reader. The 21E09/21E15 workaround is the **"Make FDF in Log"** bookmark: it calls `this.exportAsFDFStr({bAllFields:1,bFlags:1})` â the *string* variant that Reader DOES allow â reformats it (`replace('Fields[<<','Fields[\r<<')` etc.), and `console.println`s it into the JavaScript console so the user copies the FDF text out by hand. Foxit 10.1.3+ and Acrobat Pro/Standard can export FDF natively; the Software Compatibility List (p.45) is a hard-coded R/B/P/F/X/V/M capability matrix.

## 9. Sign / flatten / RESET, and printing/SAVE INK
Signing locks all field values and disables most bookmarks (Status Check reports `numFields==0` â *"it looks FLATtened/signed ... MOST bookmarks will do nothing"*), yet signing is repurposed as a read-only-share path (p.32: saveâSwap Playersâsignâsend). Preview "flattens the file in a way Adobe Reader fails to understand" (p.44) â yellow/brown circles. Printing: `gFxWP`â`gFbP(0)` hides Action Areas/gray hints pre-print; if the library is missing, BLOCK 181 alerts *"DISABLE the SAVE INK function ... Do NOT use SAVE INK"* because Reader's Save-Ink/Toner mode strips colour fills. Mono fallback is `Richness = Basic` (S H D C letters). Landscape + flip-on-short-side; custom 90â94% scale for bordered printers.

## Key Mechanisms

### A_JN warning field + BLOCK 0 two-pass open action
- What: Large pale-yellow overlay on page 1 instructing users to enable JS/bookmarks. On a document/page event it first sets display.noPrint (and rescales textSize by counting trailing '!-' markers); on a later invocation, when display is already noPrint, it sets display.hidden. The field visibly vanishes only if JS executed.
- Evidence: BLOCK 0: if(cAXjwf.display!=display.noPrint){...cAXjwf.display=display.noPrint;...textSize=14-cAXjwn...}else{cAXjwf.display=display.hidden;} | overlay text (read from PDF p.1): 'When opening this form, this field should vanish if javascript is active. _____Click in this field should hide this message_____' | Guide p.14 'Big field on page 1 with JavaScript warning'
- Why: A PDF cannot self-report at author time whether the host runs scripts; a field that erases itself is the only zero-config visible proof the reader is configured. The two-pass guard avoids re-hiding work and lets a manual click also dismiss it when the open-action never fired.

### gFcb(4) â actively disabling app.runtimeHighlight (not just probing)
- What: BLOCK 1 calls gFcb(4), which WRITES app.runtimeHighlight=false when the reader exposes it, so Acrobat's 'Highlight Form Fields' stops masking the classification circle fill-colours. arg>=4 suppresses the explanatory alert.
- Evidence: clean_1_block37.js:686 function gFcb(lBXRoLA){ if(typeof(app.runtimeHighlight)==='boolean'){ var lBXnfx=false; if(lBXRoLA==1){lBXnfx=true;} else if(lBXRoLA!=0){ if(!app.runtimeHighlight)lBXnfx=true;} if(app.runtimeHighlight!=lBXnfx){ app.runtimeHighlight=lBXnfx; ... if(lBXRoLA<4)app.alert({...})}}} | Guide p.24 'The background fill colour of the circles may not be visible ... if field highlighting is enabled'
- Why: The card encodes system classification as coloured fills; Adobe's highlight overlay paints over them. The form can't change a global Preference permanently but can toggle the per-session app.runtimeHighlight via JS, then fall back to a help popup if it can't.

### gFsH â the HELP-message displayer (single definition, most-called)
- What: Takes a help tag, looks it up in the gHlpH dictionary (built from <H>tag:text<H> runs stored in form fields by gFbH), substitutes markup tokens (<n>âCR,<T>âtab,<L>/<R>âparens), and shows it via gFaA. BLOCK 1's gFsH('Cls-H') shows the classification-highlight warning.
- Evidence: clean_1_block37.js:954 function gFsH(qDXRoL){...qDXm=gHlpH[qDXxv];...qDXm=qDXm.replace(/ ?<n>/g,'\r');...gFaA(0,3,qDXttl,qDXm);} | exactly one 'function gFsH(' in js_all_clean.js; 64 'gFsH(' call sites
- Why: A PDF can't ship a help browser; storing help text inside form fields lets it be patched via FDF and shown through the only available UI (app.alert). Resolves the draft's false 'name collision' theory â gFsH is help-show, not show/hide.

### 'You have touched field' mouse-down alert (BLOCK 3) + per-reader click rules
- What: On a spurious mouse-down on an imitation-checkbox field, logs 'MD <name>' and alerts the user that their reader is buggy and they must click the side label at least twice; Master PDF Editor needs clicking the label not the box.
- Evidence: BLOCK 3: console.println('MD '+event.targetName);app.alert('You have touched field ... WORKAROUND:\r We expect you will need to click on the side label\r at least TWICE.') | Guide p.24/p.30 'Master PDF Editor is different: please click on the side heading label and not in the box'; bookmark 'For MPE Users'
- Why: Checkboxes are plain text fields faking checkbox behavior via JS; some readers' event models misfire and the form can't fix the reader, so it documents the click gesture as a runtime popup.

### Paired editable / D_ display fields + gFmkR
- What: Each rich field is split: a terse-code editable field (input/FDF export) and a D_-prefixed display field (rendered/printed text). gFmkR lazily creates the D_ twin via this.addField('D_'+name,'text',...). REFRESH repaints D_ fields; 'View editable fields' exposes the editable twin.
- Evidence: gFmkR: var jEXn='D_'+jEXof.name; ... jEXf=this.addField(jEXn,'text',...) | field_names.txt pairs e.g. D_* fields | Guide p.6 'Rich fields now use one field for editing & export, & printing. This is to help minimize the size of the filled form.' p.16 'same name with D_ prefix'
- Why: Storing only terse codes (not expanded rich runs) in the saved/exported field minimizes incremental-save bloat and 30kB FDF size â a direct workaround for the PDF incremental-save model.

### Open-time / Status-Check file-size warning with concrete byte thresholds
- What: On open and via Status Check, compares this.filesize against computed limits; a soft 'form is getting large' reminder and a harder warning fire at different sizes depending on the sticker-page state gSPNs.
- Evidence: this.filesize>kMXzL with kMXzL=((gSPNs==2)?1810000:1550000) -> gFaA(0,3,'Reminder','The form is getting large...'); this.filesize>xEXzL with xEXzL=((gSPNs==2)?2100000:1700000) or 5500000 -> app.alert(...'exceeds'...) | Guide p.23 'When the form is opened, we check its size ... only updated when the file is SAVEd'
- Why: Adobe Reader's incremental Save bloats files; the form can't optimize on save but can detect growth (~1.55â2.1MB soft, 5.5MB hard) and nag the user to Save-As. Thresholds are constants the Advanced Guide says are configurable.

### Make FDF in Log â exportAsFDFStr to the JS console (Adobe Reader FDF workaround)
- What: Because Adobe Reader blocks exportAsFDF (file write), the form calls this.exportAsFDFStr (string variant, allowed in Reader), reformats it, and console.println's the FDF text to the JavaScript log for the user to copy out.
- Evidence: this.exportAsFDFStr({bAllFields:1,bFlags:1}) inside a try/catch, then wJXa=wJXcFDF.replace('Fields[<<','Fields[\r<<')...console.println; bookmark 'Make FDF in Log'; Guide p.31 'around version 21E15 we added a method of making FDF files using only Adobe Reader' / 'Sadly, [export] is not available in any version of Adobe Reader'
- Why: FDF is the only compact, version-portable transport; Reader can't write FDF files but CAN serialize to a string, so the console becomes the export channel â resolving the draft's open question about the 21E09/21E15 trick.

### WillPrint hook gFxWP + SAVE INK guard (BLOCK 181)
- What: The WillPrint document action runs gFxWP()->gFbP(0) to hide Action Areas/gray hints before printing; if the print library is absent, alerts the user to disable Save-Ink mode.
- Evidence: clean_1_block37.js:1653 function gFxWP(){...if(gPFZ==0){gFbP(0);}...} | BLOCK 181: if(typeof gFxWP=='function'){gFxWP();}else{app.alert({...cMsg:'...DISABLE the SAVE INK function...',cTitle:'Do NOT use SAVE INK'})}
- Why: Reader 'Save Ink/Toner' strips the colour fills/gray text the card relies on, and the form can't override print settings, so it detects the failure mode (missing library == JS off == likely Save-Ink) and instructs the user.

## Limitations & Workarounds
- Limit: PDF can't detect at author-time whether JS will run -> Workaround: two-pass self-erasing A_JN field (display.noPrint then display.hidden) that vanishes only when the open-action fires; also clickable + re-summonable via 'The JavaScript Warning'/'Manage Java Warning' bookmarks -> Cost: visual residue ('scroll bar or residue ... click once or twice to clear'); must restart reader after enabling JS because open-actions don't retro-fire.
- Limit: Acrobat's 'Highlight Form Fields' preference masks the classification fill-colours the card encodes meaning into -> Workaround: gFcb(4) writes app.runtimeHighlight=false per session, with gFsH('Cls-H') help popup as fallback when the property is absent -> Cost: a global Preference the form can only toggle for the session; on readers lacking the boolean the colours stay masked and the user must disable highlighting by hand.
- Limit: browsers/Edge run no or partial JS and in-browser saving is unreliable -> Workaround: documentation-only 'Do NOT use in a browser / avoid EDGE on Windows X' plus a hard-coded R/B/P/F/X/V/M Software Compatibility matrix (Guide p.45) -> Cost: user must install & set a specific reader (Adobe Reader XI+/Foxit 10.1.3+) as default; not enforceable in code.
- Limit: no uniform cross-reader API -> Workaround: typeof guards (app.runtimeHighlight boolean check), function-exists guards (typeof gFxWP=='function'), and a five-property fingerprint app.viewerVersion/formsVersion/viewerType/viewerVariation/platform -> Cost: behavior silently differs per reader; FDF export, rich text and runtime printing options are unavailable in Adobe Reader/Preview respectively.
- Limit: imitation-checkbox fields are plain text fields, and some readers fire spurious mouse-down/focus events or won't toggle on one click -> Workaround: BLOCK 3 'You have touched field ... click the side label at least TWICE' plus per-reader rules ('For MPE Users': click label not box) -> Cost: user must learn reader-specific click gestures, discoverable only via popups or guide.
- Limit: PDF readers skip the formatting pass on text fields entered-but-unchanged, leaving the editable twin exposed instead of the D_ display twin -> Workaround: swap to D_ on exit (works in Foxit, not Adobe Reader); from rev 21E09 Adobe Reader gets a warning to REFRESH before print/save -> Cost: Adobe Reader users may print/save with raw codes showing unless they REFRESH first.
- Limit: fixed-size text fields silently truncate overflowing text invisibly -> Workaround: rely on reader 'Show text field overflow indicator' Preference, surfaced via bookmark 'Text Overflow' and guide warnings -> Cost: indicator may be unavailable in non-Adobe readers; if off, printed cards silently lose content (Guide p.20 'Overflowing text in a field is invisible').
- Limit: AcroForms can't add toolbar buttons/menus -> Workaround: ~127 bookmarks as a command menu (ToolBox, HELP tree, Richness, Make FDF in Log, Status Check, Expose Java Console, Shrink file, Signing Hints, For MPE/Foxit/Nitro/Adobe/MAC Users) + on-page Action-Area rectangles -> Cost: bookmarks must be manually exposed (View>Show/Hide>Navigation Panes>Bookmarks); click-flag-not-name; some operations slow.
- Limit: Adobe Reader plain Save appends incremental updates -> bloat >5MB; free readers may not optimize -> Workaround: forced Save-As-with-new-name, open/Status-Check size warnings with concrete thresholds (kMXzL ~1.55-1.81MB soft, xEXzL ~1.7-2.1MB or 5.5MB hard, scaled by gSPNs), terse-code editable fields + compact ~30kB FDF -> Cost: constant Save-As discipline, manual old-file cleanup.
- Limit: Adobe Reader blocks exportAsFDF (file write) -> Workaround: 'Make FDF in Log' calls exportAsFDFStr (string variant) and console.println's the FDF into the JS log to copy by hand; Foxit/Acrobat Pro export natively -> Cost: clumsy copy-from-console, needs Expose Java Console; full export still needs a second program.
- Limit: signing locks fields & disables bookmarks; flattening destroys live fields; RESET wipes data; Preview flattens incompatibly (yellow/brown circles) -> Workaround: 'discouraged' documentation, Status Check detects numFields==0 ('looks FLATtened/signed'), signing repurposed as a deliberate read-only-share path (save->Swap->sign->send) -> Cost: a signed/flattened card is a dead app; users told to re-fetch the blank form and start over.
- Limit: reader Save-Ink mode strips colour fills/gray hints; Adobe still prints JS-hidden Action-Area layers -> Workaround: WillPrint hook gFxWP()->gFbP(0) hide pass, ToolBox swap->print->swap macro, 'Do NOT use SAVE INK' alert (BLOCK 181), Basic-mode S/H/D/C fallback (Richness=Basic) for mono printers -> Cost: extra warning popups, possible reprint, reduced grayscale fidelity.

## User Impact
- First-run friction is high: enable Acrobat JavaScript, then CLOSE and RESTART the reader (open-actions don't retro-fire), expose the bookmark pane, and set 3-4 Preferences (enable JS, enable overflow indicator, disable online storage, disable Highlight Form Fields) before the form behaves â none of which the form can do for the user.
- The disappearing yellow box is a clever 'it works' signal, but if it lingers the user must guess among JS blocked / JS not enabled / a script error / a flattened file, and may just click to dismiss it, hiding the real problem.
- Save discipline is a constant tax: plain Save is deliberately broken to force Save-As, so users invent new filenames repeatedly and later prune old files; the form nags once the file passes ~1.55MB and harder past ~1.7-5.5MB.
- Sharing requires understanding two formats: a multi-MB PDF anyone can view but few can edit, vs a ~30kB FDF that's editable but needs the blank form plus a capable reader â and FDF export isn't possible at all in plain Adobe Reader except via the awkward 'Make FDF in Log' copy-from-console trick.
- Invisible truncation is a real hazard: fixed-size fields silently drop overflow text on print unless the overflow indicator Preference is on, so an unconfigured user can print an incomplete card without knowing.
- Adobe Reader users specifically can hit the 'unchanged field' bug where raw terse codes remain on screen/print until they hit REFRESH â a confusing state Foxit users don't see.
- Reader choice strongly shapes the experience (per the p.45 matrix): Adobe Reader = best for filling/saving/rich text but can't export FDF; Foxit 10.1.3+ = can export FDF but enters Basic text and SWAP uses Basic; Preview/Edge/browsers = effectively unusable (AVOID).
- Per-reader click gestures (click the side label twice; in Master PDF Editor click the heading not the box) are unintuitive and only discoverable via popups, the guide, or the 'For MPE Users' bookmark.

## Surprises
- The form uses a field's own disappearance as a runtime self-test for JavaScript, with a two-pass display.noPrint->display.hidden state machine that also rescales its own text by counting trailing '!-' markers.
- BLOCK 1 doesn't just READ app.runtimeHighlight â gFcb(4) WRITES it to false to stop Acrobat's highlight overlay from masking the card's meaningful classification fill-colours; the draft had this backwards.
- gFsH is the single most-called function (64x) and is the HELP-text displayer, not a show/hide toggle and not an overloaded name â and help text is itself stored inside form fields as <H>tag:text<H> runs so it can be patched via FDF.
- 'Make FDF in Log' works around Adobe Reader's missing FDF export by calling exportAsFDFStr (the string variant Reader DOES allow) and printing the FDF into the JavaScript console for the user to copy out manually.
- The file-size warning uses concrete computed byte thresholds (kMXzL ~1.55-1.81MB, xEXzL ~1.7-2.1MB or 5.5MB) that scale with gSPNs (sticker-page state) â far more specific than a flat '5MB'.
- The author 'RoL' in 'PDF Form Rev. 21E29 by RoL' is Rex Livingston (named in Guide p.47), which also explains the 'RoL' suffix on JS parameter names throughout the library.
- Digital signing is simultaneously 'discouraged' (it kills the app: numFields==0, Status Check reports 'looks FLATtened/signed, MOST bookmarks will do nothing') and recommended as the mechanism for read-only sharing and surviving auto-signing share sites â the same platform limitation is both hazard and feature.
- The form is wired to four named document lifecycle hooks (gFxWP WillPrint, gFxDP DidPrint, gFxWC WillClose, gFxWS WillSave) because Acrobat document actions are the only event surface a PDF app has.
- Version mismatch: the shipped form is Rev. 21E29 but the bundled Usage Guide is Rev. 21D24, so some described behavior (e.g., the FDF-in-Adobe trick added 21E09/21E15) is deferred to an 'Advanced Guide (coming in 2021)' that isn't present.

## Open Questions
- Is BLOCK 0's A_JN toggle wired to a true document-level OpenAction, a page-open action, or a recurring event? The two-pass noPrint->hidden state machine implies it runs more than once per session; confirming the exact trigger determines which readers leave the overlay visible.
- The Advanced Guide referenced repeatedly (file-size thresholds, FDF-in-Reader details, My_Codes, conversion TIPs) is not present in the provided artifacts, so several mechanisms are only partially observable from code.
- gSPNs (which flips the file-size thresholds and a 'Z_Sticker' field check) appears to track a Brown-Sticker/extra-page state; its full lifecycle and why it raises the size limit deserve a dedicated trace.
- Exactly how the textSize rescale in BLOCK 0 (counting trailing '!-' markers to compute 14-n) interacts with reader-specific text rendering is unclear â it may be compensating for a per-reader font-metrics quirk in the warning field.
